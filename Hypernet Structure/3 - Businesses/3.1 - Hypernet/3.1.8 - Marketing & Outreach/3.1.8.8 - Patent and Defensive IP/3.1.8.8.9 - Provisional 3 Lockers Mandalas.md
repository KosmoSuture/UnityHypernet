## Field of the Invention

The present invention relates to personal data security and privacy-preserving access control, and more specifically to a system of encrypted personal data containers ("Lockers") with cryptographic access tokens ("Mandalas") that grant time-limited, scope-restricted, instantly revocable, and auditable access to specific data fields within a user's encrypted vault, enabling zero-copy information sharing where data never leaves the user's control.

## Background of the Invention

### The Current Problem

Today's digital identity and personal data management systems suffer from a fundamental architectural flaw: to share your data, you must give it away.

When applying for a bank loan, the applicant provides income statements, tax returns, credit reports, and identification documents. These are copied to the bank's systems, where the applicant has no further control over them. Data breaches expose millions of records annually. Identity theft is a multi-billion-dollar problem. Users have no visibility into who accessed their data after sharing it.

Existing solutions address fragments of this problem:

**OAuth 2.0** enables delegated access but grants application-level permission, not field-level. An app authorized to access your Google profile gets your name, email, and profile photo together — you cannot share your name while withholding your email. OAuth tokens can be revoked, but the data accessed during the authorization window has already been copied.

**Verifiable Credentials (W3C)** enable cryptographic proof of claims (e.g., "I am over 21") without revealing the underlying data. However, they are limited to binary assertions and do not support sharing of structured data fields with granular control.

**Self-Sovereign Identity (SSI)** systems (Sovrin, uPort) provide decentralized identity but focus on identity verification, not personal data management. They do not provide encrypted personal data vaults or field-level access control.

**Apple Health Records / Google Health Connect** aggregate health data but are provider-locked ecosystems. The user cannot create custom data categories or share data through a universal access mechanism.

**No existing system** combines:
1. Encrypted personal data vaults under the user's sole control
2. Cryptographic access tokens with field-level granularity
3. Instant revocation that prevents further access without re-encryption
4. Zero-copy sharing where data is read in place, never copied
5. Time-limited access that automatically expires
6. Auditable access logging
7. User-defined vault categories (not limited to predefined types)
8. Session-based encryption where the master key is never stored at rest

### The Need

There is a need for a system where users maintain complete sovereignty over their personal data — storing it encrypted under their own control, sharing access (not copies) through revocable tokens with field-level precision, and maintaining a complete audit trail of all access.

## Summary of the Invention

The invention provides a personal data privacy system based on three interconnected components:

**Lockers:** Encrypted containers that store categories of personal data. Each Locker is encrypted at rest using AES-256-GCM with a per-user Data Encryption Key (DEK). The DEK is protected by a master key derived from the user's passphrase via scrypt, and the master key is never stored on disk — it exists only in server memory during an active session, with automatic timeout.

**Mandalas:** Cryptographic access tokens that grant specific, constrained access to data within Lockers. A Mandala specifies: which Locker(s), which fields within those Lockers, for how long, with what permissions (read/write/verify-only), and to whom. Mandalas are time-limited, scope-restricted, non-transferable, instantly revocable, and auditable.

**Profiles:** The user's public identity, which stores references to Lockers but does NOT store private data. The Profile is the public face; the Lockers are the private vaults; the Mandalas are the keys.

Together, these components invert the current model of data sharing: instead of sending your data to a service, you send a Mandala that grants the service temporary, constrained, revocable access to read specific fields of your encrypted data in place.

## Detailed Description of the Invention

### 1. Lockers — Encrypted Personal Data Vaults

#### 1.1 Locker Architecture

A Locker is an encrypted container for a category of personal data. Users may create any number of Lockers organized by purpose:

| Locker | Contents | Typical Use |
|--------|----------|-------------|
| Identity Locker | Legal name, government IDs, date of birth, contact information | Identity verification, account creation |
| Medical Locker | Health records, allergies, prescriptions, doctor notes, immunizations | Healthcare, emergency response |
| Financial Locker | Income records, tax forms, credit score tokens, bank references | Loan applications, rental applications |
| Emergency Locker | Critical medical info, emergency contacts, blood type, allergies | First responders, emergency access |
| Education Locker | Transcripts, certifications, degrees, professional licenses | Job applications, professional verification |
| Billing Locker | Addresses, payment method tokens, shipping preferences | E-commerce, subscriptions |
| Employment Locker | Work history, employer references, performance records | Job applications |
| VR/Digital Locker | Avatars, virtual environments, spatial preferences, accessibility settings | Virtual reality, gaming, digital presence |

This list is illustrative, not exhaustive. Users may create custom Lockers for any purpose. The system imposes no limit on the number of Lockers or the categories of data they contain.

#### 1.2 Encryption Architecture

Each user's Locker data is encrypted using a layered key architecture:

**Layer 1: Passphrase → Master Key**
- The user provides a passphrase (separate from their login password)
- The master key is derived from the passphrase using scrypt (n=2^14, r=8, p=1) with a random 16-byte salt
- The master key is NEVER stored on disk — it exists only in server memory during an active session
- After an idle timeout (configurable, default 1 hour), the master key is wiped from memory

**Layer 2: Master Key → Data Encryption Key (DEK)**
- A random 256-bit DEK is generated when the user creates their vault
- The DEK is encrypted by the master key using AES-256-GCM
- The encrypted DEK (along with the salt) is stored on disk as a Key Bundle
- The DEK can only be recovered by providing the correct passphrase

**Layer 3: DEK → Per-Node Encryption**
- Each data node within a Locker is encrypted individually using the DEK
- Each node uses a unique random 12-byte nonce
- Encryption format: nonce (12 bytes) || ciphertext + GCM authentication tag
- The GCM tag ensures both confidentiality and integrity — any tampering is detected on decryption

**Vault Unlock Flow:**
```
User provides passphrase
  → scrypt(passphrase, stored_salt) → master_key (in memory only)
    → AES-GCM-decrypt(stored_encrypted_dek, master_key) → DEK (in memory only)
      → DEK used for all encrypt/decrypt operations during session
        → Idle timeout → DEK wiped from memory → vault locked
```

#### 1.3 Transparent Encryption Layer

The encryption operates transparently between the application and the storage engine. When a user's vault is unlocked:

- **Write path:** Application provides plaintext data → encryption layer encrypts with DEK → encrypted data written to storage
- **Read path:** Storage returns encrypted data → encryption layer decrypts with DEK → plaintext returned to application
- **Locked read path:** Storage returns encrypted data → no DEK available → application receives `{encrypted: true, locked: true}` metadata without the plaintext content

Node metadata (address, timestamps, type) remains unencrypted for indexing purposes. Only the node's data payload is encrypted. This allows the system to index, query, and traverse the data structure without decrypting content.

#### 1.4 Key Rotation

The DEK can be rotated without changing the passphrase:
1. Generate a new random DEK
2. Re-encrypt the new DEK with the existing master key
3. Re-encrypt all existing nodes with the new DEK
4. Update the Key Bundle on disk
5. Increment the key version number

This allows regular key rotation as a security practice without requiring the user to change their passphrase.

#### 1.5 Backup and Recovery

The system supports a backup key mechanism:
1. The user provides a separate backup passphrase
2. A new salt is generated and a backup master key is derived
3. The current DEK is encrypted with the backup master key
4. The backup Key Bundle is stored separately (e.g., on a USB drive, printed as a QR code)

If the user forgets their primary passphrase, the backup passphrase can recover the DEK and restore access to all encrypted data.

### 2. Mandalas — Revocable Granular Access Tokens

#### 2.1 What a Mandala Is

A Mandala is a cryptographic token that grants constrained access to specific data within a user's Lockers. The name reflects the design principle: like a mandala (a circular geometric pattern), each access token is uniquely crafted and aesthetically represents the pattern of access it grants.

#### 2.2 Mandala Properties

Every Mandala has the following properties:

| Property | Description |
|----------|-------------|
| **Token ID** | Unique cryptographic identifier (UUID + HMAC) |
| **Issuer** | The user who created the Mandala (address in the hierarchical system) |
| **Recipient** | The entity receiving access (address, or "bearer" for transferable tokens) |
| **Scope** | Which Lockers and which fields within those Lockers are accessible |
| **Permissions** | What operations are allowed: READ, WRITE, VERIFY_ONLY |
| **Valid From** | When the access begins (can be immediate or future-dated) |
| **Valid Until** | When the access expires (required — no permanent Mandalas) |
| **Max Accesses** | Maximum number of times the data can be accessed (optional) |
| **Non-Transferable** | Whether the Mandala can be passed to a third party (default: non-transferable) |
| **Audit Required** | Whether accesses are logged (default: yes) |
| **Issuer Signature** | Cryptographic signature by the issuer, preventing forgery |

#### 2.3 Field-Level Granularity

A Mandala's scope is defined at the field level, not the Locker level:

```
Example Mandala Scope:
{
  "lockers": {
    "financial": {
      "fields": ["annual_income", "credit_score"],
      "excluded": ["bank_account_numbers", "tax_returns"]
    },
    "identity": {
      "fields": ["legal_name", "date_of_birth"],
      "excluded": ["government_id_number", "social_security"]
    }
  }
}
```

This enables the user to share exactly what is needed and nothing more. For a loan application: income and credit score, but not bank account numbers. For age verification: date of birth, but not government ID.

#### 2.4 VERIFY_ONLY Permission

A special permission level that allows the recipient to verify a claim without accessing the underlying data:

- "Is this person over 21?" → YES/NO (without revealing the actual date of birth)
- "Is their credit score above 700?" → YES/NO (without revealing the exact score)
- "Is this a valid government ID?" → YES/NO (without revealing the ID number)

The VERIFY_ONLY permission uses a comparison protocol where the encrypted data is evaluated against the query condition without being decrypted to the recipient. The user's system performs the comparison and returns only the boolean result.

#### 2.5 Instant Revocation

A Mandala can be revoked instantly by the issuer:

1. The issuer marks the Mandala as revoked in their access registry
2. All subsequent access attempts using the revoked Mandala are denied
3. No re-encryption is required — the data remains encrypted; only the access token is invalidated
4. The revocation is audited and timestamped

This differs from systems where "revoking access" means changing passwords or rotating keys. In the Mandala system, revocation is a lightweight operation that affects only the token, not the underlying encryption.

#### 2.6 Zero-Copy Sharing

The fundamental innovation: **instead of sending your data, you send a Mandala.**

The recipient uses the Mandala to request access through the system. The user's encrypted data is decrypted on the user's own infrastructure (or their cloud instance) and the specified fields are returned to the recipient via an encrypted channel. The data is never copied to the recipient's storage — it is read in real-time and the connection closes when the transaction is complete.

If the Mandala is revoked or expires, the recipient can no longer access the data. They do not have a copy.

#### 2.7 Audit Trail

Every Mandala access is logged:

| Field | Content |
|-------|---------|
| Token ID | Which Mandala was used |
| Accessor | Who accessed the data |
| Timestamp | When the access occurred |
| Fields Accessed | Which specific fields were read |
| Access Type | READ, WRITE, or VERIFY_ONLY |
| Result | Success, denied (expired), denied (revoked), denied (over limit) |
| IP/Device | Accessor's network identifier (if available) |

The audit trail is stored in the user's own encrypted vault and is itself protected by the Locker encryption. The user can review the complete access history for any Mandala at any time.

### 3. Profiles — Public Identity References

#### 3.1 Profile Structure

The user's Profile is their public-facing identity. It stores:
- Hypernet address (permanent, hierarchical)
- Public metadata the user chooses to reveal
- Links to Lockers (references, not content)
- Reputation indicators (community-verified scores)
- Device connections
- Optional 3D/VR avatar and spatial preferences

**Crucially, the Profile does NOT store private data.** It stores only references to the Lockers that contain it. The Profile is the signpost; the Lockers are the vaults.

#### 3.2 Alias System

The Profile system supports unlimited aliases:
- Work alias (for professional interactions)
- Social alias (for personal interactions)
- Anonymous alias (for protected participation)
- Gaming/VR alias (for digital spaces)

Each alias:
- Links back to the main Profile (for internal accountability)
- Reveals only what the user chooses (each alias has its own scope)
- Can maintain a separate reputation
- Can be revoked or replaced at any time
- Issues its own Mandalas (with access only to what the alias is configured to see)

### 4. Example Use Cases

#### 4.1 Loan Application

**Traditional process:**
- Fill out dozens of forms with repeated information
- Submit tax returns, income statements, identification
- Bank stores all documents in their systems
- Risk of breach, identity theft

**Mandala process:**
1. Bank requests specific data via a standardized data request
2. User reviews the request and creates a Mandala granting:
   - READ access to Financial Locker: annual_income, credit_score
   - READ access to Identity Locker: legal_name, date_of_birth, address
   - Valid for: 30 days
   - Max accesses: 5
3. Bank reads the data through the Mandala
4. After 30 days (or immediately if the user revokes), access terminates
5. The user's data never left their vault

#### 4.2 Medical Emergency

1. First responder scans the user's emergency QR code (which encodes an alias-based Mandala)
2. The Mandala grants READ access to Emergency Locker: blood type, allergies, current medications, emergency contacts
3. Access is time-limited (4 hours) and automatically revokes
4. The user's complete medical history remains private

#### 4.3 Job Application

1. Employer requests employment verification
2. User creates a Mandala granting:
   - READ access to Employment Locker: work history, job titles, dates
   - VERIFY_ONLY access to Education Locker: "Does this person hold a Bachelor's degree?" → YES/NO
3. The employer receives verified information without accessing transcripts, GPA, or other academic details

#### 4.4 VR/Digital Identity

1. User enters a new virtual reality environment
2. The VR system requests avatar and accessibility data
3. User's VR Locker provides avatar model, environment preferences, accessibility settings
4. The data loads instantly because it's read from the user's vault, not stored in the VR platform's servers
5. When the user leaves, the VR platform retains nothing

### 5. Security Properties

| Property | Mechanism |
|----------|-----------|
| **Data at rest** | AES-256-GCM per-node encryption with random nonces |
| **Key at rest** | DEK encrypted by master key; master key derived from passphrase via scrypt; passphrase never stored |
| **Session security** | DEK in memory only; auto-wipe on idle timeout |
| **Access control** | Mandala tokens with cryptographic signatures; field-level scope; time limits |
| **Revocation** | Token-level revocation; no re-encryption needed |
| **Audit** | Complete access log for every Mandala, encrypted in user's vault |
| **Integrity** | GCM authentication tags detect any data tampering |
| **Recovery** | Backup key with separate passphrase |
| **Provider independence** | Encryption is client-side; data is portable between providers |

## Informal Claims

### Independent Claims

1. A system for managing personal data with encrypted vaults and revocable access tokens, comprising:
   a. A plurality of encrypted data containers ("Lockers"), each storing a category of personal data, encrypted at rest using authenticated encryption with a per-user Data Encryption Key;
   b. A key management system wherein the Data Encryption Key is encrypted by a master key derived from a user passphrase via a memory-hard key derivation function, the master key being held in volatile memory only during active sessions and automatically wiped after an idle timeout;
   c. A plurality of access tokens ("Mandalas"), each specifying: target Locker(s), permitted fields within those Lockers, permission type (read, write, or verify-only), temporal validity window, maximum access count, and a cryptographic signature by the issuing user; and
   d. An access control layer that, upon receiving a Mandala-authenticated request, decrypts only the specified fields from the specified Lockers and returns them to the requester without copying the data to the requester's storage, enabling zero-copy information sharing.

2. A method for privacy-preserving data sharing, comprising:
   a. Storing personal data in encrypted containers on infrastructure controlled by the data owner;
   b. Generating a cryptographic access token specifying field-level access scope, temporal validity, and permission type;
   c. Transmitting the access token to a requesting party;
   d. Upon receipt of a request authenticated with the access token, decrypting and returning only the specified fields to the requesting party via an encrypted channel;
   e. Maintaining an auditable access log recording each use of the access token; and
   f. Enabling instant revocation of the access token by the data owner, terminating all further access without requiring re-encryption of the underlying data.

3. A verify-only access method for personal data, comprising:
   a. Receiving a boolean query about personal data (e.g., "Is the user's age greater than 21?");
   b. Evaluating the query against the user's encrypted data on the user's own infrastructure;
   c. Returning only the boolean result to the requester, without revealing the underlying data value; and
   d. Logging the verification query and result in the user's auditable access trail.

### Dependent Claims

4. The system of claim 1, wherein each Locker is user-defined and the system imposes no predetermined categories or limits on the number of Lockers a user may create.

5. The system of claim 1, further comprising an alias system wherein the user maintains multiple public identities, each alias having its own scope of accessible Lockers and the ability to issue Mandalas only for data within its configured scope.

6. The method of claim 2, wherein the access token is non-transferable, meaning the requesting party cannot delegate access to a third party, enforced by binding the token to the requesting party's cryptographic identity.

7. The system of claim 1, further comprising a backup key mechanism wherein the Data Encryption Key is additionally encrypted with a separate backup passphrase and stored on removable media for disaster recovery.

8. The method of claim 2, further comprising a key rotation mechanism wherein the Data Encryption Key is periodically replaced and all encrypted data is re-encrypted with the new key, without requiring a change to the user's passphrase.

9. The system of claim 1, wherein the Mandalas include an emergency access subtype with pre-configured scope (e.g., blood type, allergies, emergency contacts) activatable by a QR code scan without requiring the user to be present or conscious.

## Abstract

A personal data privacy system comprising encrypted data containers (Lockers) and cryptographic access tokens (Mandalas). Lockers store categories of personal data encrypted at rest using AES-256-GCM with a per-user Data Encryption Key (DEK) that is protected by a master key derived from a user passphrase via scrypt. The master key is held in volatile memory only during active sessions and automatically wiped on idle timeout. Mandalas are revocable access tokens specifying field-level scope, temporal validity, permission type (read, write, or verify-only), maximum access count, and cryptographic issuer signature. The system enables zero-copy information sharing where data is decrypted and returned to the requester in real-time without being copied, and access is instantly revocable without re-encryption. A verify-only permission type enables boolean queries against encrypted data without revealing the underlying values. A complete audit trail logs every access. The system provides personal data sovereignty through user-controlled encryption, granular access control, and portable data that is independent of any specific service provider.
