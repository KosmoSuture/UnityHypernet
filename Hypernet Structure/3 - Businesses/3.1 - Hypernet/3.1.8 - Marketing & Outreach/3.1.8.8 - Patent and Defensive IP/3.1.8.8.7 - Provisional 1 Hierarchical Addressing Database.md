## Field of the Invention

## Background of the Invention

### Problems with Existing Approaches

Modern database systems for representing complex, interconnected data face several fundamental limitations:

**Relational databases** (PostgreSQL, MySQL) decompose entities into flat tables with foreign key relationships. Representing hierarchical or graph-structured data requires multiple join operations, which become expensive at scale. The schema must be defined separately from the data, creating a dual-maintenance burden.

**Graph databases** (Neo4j, SurrealDB, Amazon Neptune) store entities as nodes and relationships as edges. They support arbitrary connections but rely on opaque identifiers (UUIDs, auto-increment IDs) that carry no semantic information. Querying for "all entities in category X" requires either label-based filtering or traversal, neither of which benefits from the native key ordering of the storage engine.

**Hierarchical databases** (LDAP, some XML databases) support tree structures but typically lack arbitrary cross-links between nodes at different levels of the hierarchy. They enforce a single tree, whereas real-world data exhibits both hierarchical structure and non-hierarchical relationships.

**Document databases** (MongoDB, CouchDB) support nested documents with dot-notation field access (e.g., `address.city`), but the dot notation is a field path within a document, not a primary key addressing scheme. There is no concept of the address encoding semantic category membership.

**Hypergraph databases** (HyperGraphDB) support relationships of arbitrary arity but use opaque identifiers for nodes and do not implement hierarchical addressing as a native primitive.

**No existing system** combines hierarchical addressing, graph-structure (arbitrary cross-links), self-describing schema (where the address IS the schema), and storage-engine-level query optimization for hierarchical prefix operations in a single architecture.

### The Need

There is a need for a data storage and retrieval system that:
1. Assigns each data node a hierarchical address that encodes its categorical position
2. Uses that address as the native primary key, enabling O(log N) subtree queries via prefix matching
3. Supports arbitrary typed relationships (links) between any nodes, regardless of their position in the hierarchy
4. Eliminates the need for a separate schema definition layer by reserving a metadata address space within the hierarchy itself
5. Supports unlimited expansion without schema migration or structural changes

## Summary of the Invention

The invention provides a graph database system in which every data node is assigned a hierarchical dot-notation address (e.g., `1.1.3.2.1`) that simultaneously encodes:

- The node's position within a categorical hierarchy (category, subcategory, type, instance, component)
- The node's primary key in the underlying storage engine
- The node's query operand (prefix-matching enables efficient subtree retrieval)
- The node's schema context (by inheriting type definitions from a reserved metadata address space)

The system maintains a dual graph structure: (1) an implicit hierarchy tree derived from the address prefix relationships, and (2) an explicit typed link graph supporting arbitrary relationships between any nodes. This combination enables both hierarchical navigation (parent/child/sibling by address manipulation) and general graph traversal (follow typed links between unrelated nodes).

A reserved metadata address space (.0 at every level) provides self-describing schema definitions without requiring an external schema registry. The node at address N.0 defines the schema (required fields, optional fields, link types, validation rules) for all nodes under address prefix N. This pattern is infinitely recursive: N.0 defines the schema for N.*, N.X.0 defines the schema for N.X.*, and so forth. Schema evolution occurs by creating new version entries under the .0 nodes, without migrating existing data.

## Detailed Description of the Invention

### 1. Addressing System

#### 1.1 Address Format

Each node in the system is assigned a dot-separated hierarchical address comprising one or more address parts. An address part is an alphanumeric string (letters, digits, hyphens). Parts are separated by the period (`.`) character.

Examples:
```
0          — System root
0.1        — Hypernet Core
0.1.7      — AI Swarm subsystem
1.1        — A specific person (Matt Schaeffer)
1.1.3.2.1  — A specific communication item within that person's account
2.1        — An AI citizen account
```

#### 1.2 Category Encoding

The first address part defines the top-level category:

| Prefix | Category |
|--------|----------|
| 0 | System Definitions (schemas, metadata, processes) |
| 1 | People (human accounts) |
| 2 | AI Entities (AI citizen accounts) |
| 3 | Businesses and Organizations |
| 4 | Knowledge and Information |
| 5 | Object Type Definitions |
| 6 | Link Type Definitions |
| 7 | Processes and Workflows |
| 8 | Flags and Tags |
| 9 | Concepts and Ideas |

This encoding is extensible — new top-level categories can be added without modifying existing data or schema.

#### 1.3 Depth Semantics

Each subsequent address part narrows the scope:
- Part 1: Category (e.g., `1` = People)
- Part 2: Entity within category (e.g., `1.1` = Matt Schaeffer)
- Part 3: Subcategory within entity (e.g., `1.1.3` = Communications)
- Part 4: Type within subcategory (e.g., `1.1.3.2` = Chat Messages)
- Part 5+: Instance and component (e.g., `1.1.3.2.1` = Specific message)

The depth is unlimited. There is no maximum nesting level.

#### 1.4 Extended Address Notation

Addresses support resource-level resolution using a colon (`:`) separator:

```
NODE_ADDRESS:RESOURCE_PATH:SUBSECTION
```

Example: `1.1:profile.json:contact.email` resolves to the `contact.email` field within the `profile.json` resource at node `1.1`.

#### 1.5 Address Properties

Given address `A = P1.P2.P3...Pn`:
- **Parent:** `P1.P2...P(n-1)` (removing the last part)
- **Children:** All addresses matching prefix `A.` (the address followed by a period and additional parts)
- **Siblings:** All addresses sharing the same parent
- **Depth:** The number of parts (n)
- **Category:** The first part (P1)
- **Is ancestor of B:** True if B starts with `A.`

These properties are computed directly from string operations on the address, requiring no database lookups.

### 2. Storage Architecture

#### 2.1 Storage Engine

The system uses an ordered key-value storage engine (preferably LMDB, though the architecture is compatible with RocksDB, LevelDB, or any B+ tree-based store) where keys are stored in sorted order. This enables efficient prefix-scan operations.

#### 2.2 Sub-Database Layout

The storage system comprises the following logical sub-databases (implemented as LMDB named databases or equivalent):

| Sub-Database | Key | Value | Purpose |
|-------------|-----|-------|---------|
| `nodes` | Dot-notation address | Serialized node data (JSON or MessagePack) | Primary node storage |
| `links` | Link hash | Serialized link data | Relationship storage |
| `adj_from` | Source address + link hash | Empty | Outgoing link adjacency index |
| `adj_to` | Target address + link hash | Empty | Incoming link adjacency index |
| `type_idx` | Type address + node address | Empty | Type-based lookup index |
| `history` | Address + version number | Serialized node snapshot | Version history (append-only) |
| `meta` | Key string | Value | System metadata |

Additional optional sub-databases:
- `permissions`: Address prefix → permission tier
- `fulltext`: Term → set of node addresses (inverted index)
- `flags`: Flag address + node address → empty
- `blobs`: Blob hash → binary data

#### 2.3 Prefix-Scan Query Optimization

Because addresses are stored in sorted (lexicographic) order in a B+ tree, querying for all nodes under a given prefix is a single cursor-seek operation followed by sequential reads:

```
# Pseudocode: Find all nodes under prefix "1.1.3"
cursor = db.cursor()
cursor.seek("1.1.3")  # O(log N) seek
while cursor.key().startswith("1.1.3"):
    yield cursor.value()
    cursor.next()       # O(1) sequential read
```

This gives O(log N + k) performance for retrieving k nodes under a prefix from N total nodes, which is optimal for tree-structured data.

#### 2.4 Zero-Copy Reads

When using LMDB with memory-mapped I/O (mmap), read operations return direct pointers to the memory-mapped file pages. No data is copied from kernel space to user space. This provides read performance limited only by memory bus speed.

### 3. Node Data Structure

Each node in the system has the following fields:

| Field | Type | Description |
|-------|------|-------------|
| address | String (dot-notation) | Primary key — the hierarchical address |
| type_address | String (dot-notation) or null | Reference to the node's type definition (at a 0.5.* address) |
| data | Dictionary | Arbitrary key-value payload |
| created_at | ISO 8601 timestamp | Creation time |
| updated_at | ISO 8601 timestamp | Last modification time |
| deleted_at | ISO 8601 timestamp or null | Soft-delete timestamp (null if active) |
| source_type | String or null | Origin indicator ("upload", "integration", "api", "import", "ai_generated") |
| source_id | String or null | External reference identifier |
| creator | String (dot-notation) or null | Address of the creating entity |
| position_2d | {x, y} or null | 2D layout coordinates |
| position_3d | {x, y, z} or null | 3D spatial coordinates |
| flags | List of strings | Flag addresses (references to 0.8.* flag definitions) |
| is_instance | Boolean | Whether this node is an instance of its type |

Nodes are serialized to JSON (or optionally MessagePack for binary efficiency) before storage.

### 4. Link Data Structure

Links represent typed relationships between any two nodes in the system:

| Field | Type | Description |
|-------|------|-------------|
| from_address | String (dot-notation) | Source node address |
| to_address | String (dot-notation) | Target node address |
| relationship | String | Relationship type (e.g., "AUTHORED_BY", "DEPENDS_ON", "MEMBER_OF") |
| category | String (dot-notation) | Link category address (reference to 0.6.* link type definition) |
| data | Dictionary | Arbitrary relationship attributes |
| created_at | ISO 8601 timestamp | Creation time |
| updated_at | ISO 8601 timestamp | Last modification time |
| status | String | Lifecycle state ("PROPOSED", "ACTIVE", "DEPRECATED", "ARCHIVED", "REJECTED") |
| creator | String (dot-notation) or null | Address of the creating entity |
| bidirectional | Boolean | Whether the relationship applies in both directions |
| weight | Float or null | Optional relationship strength/importance |
| valid_from | ISO 8601 timestamp or null | Temporal validity start |
| valid_to | ISO 8601 timestamp or null | Temporal validity end |
| access_level | String | Required permission level to traverse this link |

Links are stored by hash (computed from from_address + to_address + relationship + created_at) and indexed in both the `adj_from` and `adj_to` sub-databases for bidirectional traversal.

### 5. The .0 Metadata Reservation Pattern

#### 5.1 Principle

For every node address N, the address N.0 is reserved for metadata about the N address space. This creates a self-describing schema system embedded within the data itself.

#### 5.2 How It Works

- Node `0.0` defines the metadata schema for the entire `0.*` (System) space
- Node `1.0` defines the metadata schema for the entire `1.*` (People) space
- Node `1.1.0` defines the metadata schema for Matt Schaeffer's (`1.1.*`) address space
- Node `1.1.3.0` defines the metadata schema for Matt's Communications (`1.1.3.*`)

#### 5.3 Standard Metadata Sub-Sections

Each .0 metadata node may contain up to 9 standard sub-sections:

| Address | Section | Content |
|---------|---------|---------|
| N.0.1 | Publishing | Visibility, access control, distribution rules |
| N.0.2 | Versioning | Version history, change tracking |
| N.0.3 | Classification | Type, category, tags, taxonomy |
| N.0.4 | Ownership | Creator, maintainers, governance |
| N.0.5 | Quality | Validation rules, integrity checks |
| N.0.6 | Relationships | Link type constraints, cardinality rules |
| N.0.7 | Presentation | Display hints, rendering preferences, spatial layout |
| N.0.8 | Integration | External system mappings, API bindings |
| N.0.9 | Extensions | Custom metadata beyond the standard sections |

#### 5.4 Infinite Recursion

The .0 pattern is infinitely recursive. The metadata node N.0 can itself have a metadata node N.0.0 that defines the schema for how metadata is structured. This recursion bottoms out at the system-level metadata definition `0.0.0`, which is the foundational schema for all metadata.

#### 5.5 Schema Evolution

Because schema definitions are nodes within the database (not external configuration), schema evolution follows the same versioning mechanism as data evolution:
1. Create a new version of the .0 node with updated field definitions
2. Existing data nodes remain valid under the previous schema version
3. New data nodes are validated against the latest schema version
4. No data migration is required — old and new schemas coexist

#### 5.6 Three-Tier Metadata Architecture

For efficiency, metadata is implemented in three tiers:

| Tier | Storage | Use Case |
|------|---------|----------|
| Tier 1: Frontmatter | Embedded in the node's data field | Essential metadata (ha, type, creator, status, flags) — always loaded |
| Tier 2: Index | Stored at N.0 | Schema definitions, validation rules — loaded on demand |
| Tier 3: Extended | Stored at N.0.1 through N.0.9 | Rich metadata — loaded only when specifically requested |

This tiered approach ensures that basic operations (read, query, traverse) only load Tier 1 metadata, while schema validation and rich metadata queries load additional tiers as needed.

### 6. Dual Graph Structure

The system maintains two simultaneous graph structures over the same set of nodes:

#### 6.1 Implicit Hierarchy Graph

The hierarchy is derived from address relationships:
- `1.1` is the parent of `1.1.3`
- `1.1.3` is the parent of `1.1.3.2`
- `1.1.3.2` is the parent of `1.1.3.2.1`

No explicit storage is required for hierarchy edges — they are computed from address string operations.

#### 6.2 Explicit Link Graph

The link sub-databases store arbitrary typed relationships between any nodes:
- `1.1.3.2.1` —[AUTHORED_BY]→ `1.1` (a message authored by a person)
- `2.1` —[REVIEWED_BY]→ `2.2` (an AI reviewed by another AI)
- `0.5.1` —[EXTENDS]→ `0.5.0` (a type extending the master type)

#### 6.3 Combined Query

Queries can combine hierarchical prefix matching with link traversal:
1. Prefix scan: Find all nodes under `1.1.3` (all of Matt's communications)
2. Link filter: Of those, find nodes linked to `2.1` via AUTHORED_BY (communications authored by AI citizen 2.1)
3. Result: Communications in Matt's space that were written by an AI

This combines the efficiency of B+ tree prefix scans (for the hierarchy) with the flexibility of graph traversal (for cross-links).

### 7. Version History

Every modification to a node creates a versioned snapshot in the `history` sub-database:

| Key | Value |
|-----|-------|
| `{address}/v{version_number}` | Serialized snapshot of the node before modification |

The version number is monotonically increasing per node. This provides:
- Complete audit trail of all changes
- Ability to reconstruct any historical state
- Compliance with data protection requirements (soft-delete only — deleted nodes remain in history)

### 8. Example Embodiment

The following illustrates a concrete use of the system:

**Scenario:** A personal data management system where a user stores contacts, documents, photos, and communications, all cross-linked and searchable.

**Address structure:**
```
1.1                     — User account
1.1.0                   — User profile schema definition
1.1.2                   — Documents
1.1.2.1                 — Tax documents
1.1.2.1.0               — Tax document schema
1.1.2.1.2026            — Tax document for 2026
1.1.3                   — Communications
1.1.3.0                 — Email (schema)
1.1.3.0.1               — Receipts (sub-schema)
1.1.4                   — Relationships
1.1.4.1                 — Family
1.1.4.2                 — Professional contacts
```

**Links:**
```
1.1.3.0.1.00042 —[REFERENCES]→ 1.1.2.1.2026  (receipt email references tax document)
1.1.4.1.3 —[AUTHORED_BY]→ 1.1.3.0.00015      (family member sent an email)
1.1.2.1.2026 —[DEPENDS_ON]→ 1.1.3.0.1.00042  (tax document depends on receipt)
```

**Query:** "Find all tax-related emails from family members"
1. Prefix scan `1.1.3.0.1` (all receipt emails)
2. Follow AUTHORED_BY links to filter by `1.1.4.1.*` (family members)
3. Follow REFERENCES links to find associated documents under `1.1.2.1` (tax documents)

### 9. Alternative Embodiments

The hierarchical addressing system described herein may be implemented using various storage engines including but not limited to:
- LMDB (Lightning Memory-Mapped Database)
- RocksDB
- LevelDB
- SQLite (using the address as a TEXT primary key with LIKE-based prefix matching)
- Any B+ tree or LSM-tree based key-value store that supports ordered key iteration

The address format may use separators other than the period (`.`) character, including but not limited to forward slash (`/`), colon (`:`), or hyphen (`-`), provided the separator character is not used within individual address parts.

The category encoding (which top-level number maps to which domain) is configurable and may be extended without modifying the core addressing or storage mechanisms.

The serialization format for nodes and links may be JSON, MessagePack, CBOR, Protocol Buffers, or any other structured serialization format.

The system may operate as a standalone database, as an embedded library within an application, or as a networked service accessible via REST API, GraphQL, or custom query protocols.

## Abstract

A graph database system and method using hierarchical dot-notation addresses as native primary keys. Each data node receives a dot-separated address encoding its position in a categorical hierarchy. The address serves simultaneously as primary key, schema context, and query operand. An ordered key-value storage engine enables efficient subtree queries via prefix-scan operations. A separate link storage subsystem maintains typed relationships between any nodes in the hierarchy, creating a dual graph structure: an implicit hierarchy from address relationships and an explicit typed link graph. A reserved metadata address space (.0 at every level) provides self-describing schema definitions within the data itself, supporting unlimited schema evolution without data migration. The system enables combined hierarchical navigation and graph traversal in a single unified database architecture.
