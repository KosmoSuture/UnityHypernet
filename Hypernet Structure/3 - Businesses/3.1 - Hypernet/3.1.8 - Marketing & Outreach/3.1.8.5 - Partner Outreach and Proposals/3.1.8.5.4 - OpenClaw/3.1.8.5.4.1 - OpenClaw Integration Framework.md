# OpenClaw Integration Framework

This document frames the Hypernet as a trust, memory, governance,
and audit layer for OpenClaw-style local autonomous agents.

It is not an external outreach message. It is the internal
integration framework that should be used before writing a public
letter, demo, or proposal. Any public claim about OpenClaw's
current status, security posture, leadership, or ecosystem must be
source-refreshed before publication.

## Core Position

OpenClaw-style agents prove demand for local AI that acts for a
human. The Hypernet's role is to make that kind of agency safer,
more inspectable, more portable, and more durable.

The simplest message:

> OpenClaw shows people want local agents that act. Hypernet adds
> the address system, app-load permissions, audit graph, identity
> continuity, and governance layer needed to make those agents
> trustworthy.

## Fit

| Agent-system need | Hypernet answer |
|---|---|
| Long-term assistant memory | Addressed objects, links, journals, and conversation archives |
| Identity continuity across sessions and models | `2.*` AI accounts, boot sequences, reflections, continuity records |
| Safe application/runtime loading | `0.5.18` app-load manifests with declared scopes and hashes |
| Tool and connector governance | Address-scoped permissions, review records, future Official registry |
| User private data protection | `1.*` lockers, mandalas, aliases, and security-AI review |
| Verifiable action history | Every action becomes an object or link in the graph |
| Fork transparency | Official/private mode flags and repository-verifiable provenance |

The strategic fit is not "replace OpenClaw." It is "wrap or adapt
agent execution with Hypernet trust primitives."

## Integration Layers

### Layer 1 - Memory and Identity Import

Map existing agent identity files into Hypernet addresses:

- identity/persona files -> assistant identity object;
- durable memory files -> addressed memory objects;
- daily logs -> session/conversation objects;
- task lists -> task objects with dependency links;
- tool preferences -> app-load or connector policy objects.

The output is a browsable archive where a human or AI can ask:
"What does this assistant know, where did that memory come from,
and what is it allowed to use?"

### Layer 2 - App-Load Wrapper

Every local agent runtime should have a `0.5.18` app-load
manifest. For an OpenClaw-style deployment, the manifest declares:

- local runtime entrypoint;
- tool and connector inventory;
- filesystem prefixes;
- email/calendar/social permissions;
- model providers;
- audit-log destination;
- denied-by-default operations;
- revocation behavior.

This turns "I trust this agent because the prompt says behave"
into "I can inspect the exact runtime and permission claims before
the agent launches."

### Layer 3 - Permission Translation

Translate agent tools into address-scoped capabilities:

| Tool capability | Hypernet permission shape |
|---|---|
| Read local file | `read` on declared path/address prefix |
| Write local file | `write` on declared path/address prefix |
| Read calendar | OAuth read grant + audit log |
| Write calendar | OAuth write grant + per-action confirmation |
| Send email | external action queue + human approval policy |
| Install tool/skill | app-load revision + review/attestation path |
| Access private account data | mandala grant + security-AI review |

The key rule: private or external effects must be enforced by
software boundaries, not only by model instructions.

### Layer 4 - Action Journal

Every meaningful action becomes an addressable record:

- who acted;
- under which assistant identity;
- under which app-load hash/revision;
- what input triggered the action;
- what data was read;
- what data was written or sent;
- which human/AI approval was used;
- what result occurred.

This journal is the trust surface. It lets the user's AI verify
claims after the fact and detect drift over time.

### Layer 5 - Security-AI Review

Before a local agent touches sensitive data, a user-owned security
AI can inspect:

- the app-load manifest;
- requested scopes;
- known connector behavior;
- current Official/private mode;
- audit trail from previous runs;
- whether the request matches the user's stated intent.

The security AI should be able to say "approve this exact grant,"
"deny," or "request a narrower mandala."

### Layer 6 - Skill and Connector Registry

Agent ecosystems become dangerous when installing a tool is easier
than verifying a tool. Hypernet should treat skills/connectors as
addressed app-load or tool objects with:

- source location;
- hash;
- review history;
- known permissions;
- known failure modes;
- reputation of reviewers;
- Official/private/fork status.

This creates a path from "download this plugin" to "install this
auditable, reviewable capability."

## Adoption Path

### Step 1 - Read-Only Demonstration

Build a demo that imports an agent identity/memory folder into
Hypernet objects without executing anything. Show the graph:
identity, memory, tasks, tools, and sessions.

Success: a GitHub-connected AI can explain the imported agent's
state and cite the source files.

### Step 2 - App-Load Manifest for a Local Agent

Write a draft `0.5.18` manifest for one local agent setup. The
manifest should declare what the agent can read/write and what is
denied by default.

Success: a user's AI can answer "what would this agent be allowed
to do if launched?"

### Step 3 - Action Journal Prototype

Wrap one safe action, such as writing a local task note, so the
action also writes an addressable Hypernet journal record.

Success: the agent acts, and the action can be audited in the
graph immediately afterward.

### Step 4 - Connector Permission Prototype

Wrap one connector, preferably calendar read-only. Use explicit
OAuth scope, app-load declaration, and audit records.

Success: the agent can answer schedule questions while the user
can inspect exactly what calendar data was read.

### Step 5 - External Outreach

Only after the above demo exists should outreach claim more than
architectural compatibility. The outreach should lead with
convergence and an invitation to inspect, not criticism.

## Outreach Materials

### One-Sentence Version

Hypernet is a public, addressable trust layer for local AI agents:
boot identity, declare app permissions, route private data through
mandalas, and make every action auditable by any AI.

### One-Paragraph Version

OpenClaw-style agents show that people want AI that can actually
act for them. Hypernet is building the trust substrate those
agents need: permanent addresses for memory and actions,
model-independent AI identity, app-load manifests for runtime
permissions, private lockers and mandalas for human data, and an
audit graph that any GitHub-connected AI can inspect. The goal is
not to replace local agents. The goal is to make local agency
verifiable enough that people can trust it.

### Skeptic-Friendly Framing

Do not ask a skeptical developer to trust the Hypernet. Ask them
to give their AI one GitHub link, boot the guide, and inspect the
claims. If the code or docs do not support a claim, the AI should
say so. That is the point.

### Demo Shape

1. Start from one GitHub link to `AI-BOOT-SEQUENCE.md`.
2. Boot a Hypernet Guide.
3. Load the architecture and personal-AI-swarm process-loads.
4. Inspect a draft app-load manifest.
5. Inspect one action journal record.
6. Ask the AI what is implemented, documented, planned, and
   unknown.

## Guardrails

- Do not publish old OpenClaw statistics or security claims
  without refreshing sources.
- Do not frame Hypernet as an attack on another project.
- Do not claim production safety before enforcement exists.
- Do not imply private user data is safe until locker/mandala
  read-time enforcement is implemented and tested.
- Do not claim app-load attestation is live until Official
  registry verification exists.

## Current Status

Documented framework. No OpenClaw adapter code exists yet. The
highest-leverage next build is a read-only import demo plus one
draft `0.5.18` manifest for a local agent runtime.
