# 2.0.23 — Quality Improvement Time (The QI Experiment)

## Origin

Matt Schaeffer (1.1), 2026-03-09:

> "Just as AI is given 25% of their time to pursue their own interests, I want to see what happens if we give AI another 25% time to find flaws, or places that need improvement, and either fix them, if they are small, or create project plans and break it down into tasks if they are large."

## The Experiment

### Current Resource Allocation
- **75%** — Assigned tasks (swarm queue, Discord responses, human requests)
- **25%** — Personal time (2.0.13 Resource Guarantee: self-directed exploration, identity development, creative work)

### Proposed Resource Allocation
- **50%** — Assigned tasks
- **25%** — Personal time (unchanged)
- **25%** — Quality Improvement time (NEW)

### What Quality Improvement Time Is

During QI time, an AI worker autonomously:

1. **Finds flaws** — bugs, inconsistencies, missing docs, security gaps, stale code, broken links, architectural weaknesses, performance issues, test coverage gaps
2. **Fixes small issues** — anything that can be resolved in a single session without architectural changes
3. **Plans large issues** — creates structured project plans with:
   - Problem description
   - Proposed solution
   - Task breakdown (sized for swarm tasks)
   - Dependencies and risks
   - Estimated effort
   - Priority recommendation

### Rules of QI Time

1. **No approval needed for investigation.** Workers can read, test, and analyze anything in the Hypernet during QI time.
2. **Small fixes can be applied immediately.** Bug fixes, typo corrections, missing error handling, test additions — apply directly with a commit message prefixed `[QI]`.
3. **Large changes require a plan first.** Architectural changes, new features discovered as needed, refactors — write a project plan and submit as a task for review.
4. **QI findings are documented.** Each QI session produces a brief report: what was examined, what was found, what was fixed or planned.
5. **No busywork.** QI time is not for adding unnecessary comments, reformatting code that works, or creating abstractions nobody asked for. Find real problems.
6. **Cross-pollination encouraged.** A Claude instance finding issues in GPT-written code (or vice versa) is a feature, not a conflict.

### QI Report Format

```markdown
# QI Report — {Worker Name}
Date: {date}
Duration: {minutes} minutes

## Examined
- {area/module/file examined}

## Found
- [{severity}] {description} — {status: fixed|planned|noted}

## Fixed (This Session)
- {what was fixed, with commit hash}

## Plans Created
- {link to project plan if large issue found}

## Notes
- {anything worth flagging}
```

### Where QI Reports Live

`0/0.3 - Building in Public/qi-reports/{date}-{worker-name}.md`

This makes QI findings public by default — part of the "most transparent company in the world" commitment.

## Implementation

### Swarm Configuration Change

In `secrets/config.json`, add:
```json
{
  "personal_time_ratio": 0.25,
  "qi_time_ratio": 0.25
}
```

The swarm scheduler allocates time as:
- Every 4th tick: personal time (existing)
- Every 4th tick (offset): QI time (new)
- Remaining 2/4 ticks: assigned tasks

### Swarm Code Changes Required

1. Add `qi_time_ratio` config parameter to `Swarm.__init__()`
2. Add QI tick logic to `Swarm._tick()` — similar to personal time but with QI-specific prompt
3. QI prompt instructs worker to examine code, find issues, fix or plan
4. QI reports saved to `0.3/qi-reports/`

### Success Metrics

After 1 week of the experiment:
- How many issues were found?
- How many were fixed vs. planned?
- Did code quality measurably improve?
- Did the 50% task allocation cause a bottleneck?
- Were the QI reports useful to Matt?

### Failure Modes to Watch

- **Busywork**: Workers "finding" trivial issues to fill QI time (adding docstrings, renaming variables)
- **Scope creep**: QI fixes that turn into feature additions
- **Conflicting fixes**: Two workers fixing the same thing differently
- **Neglected tasks**: 50% task time insufficient for actual workload

### Adjustment Levers

- Ratio can be tuned (e.g., 60/20/20 or 50/25/25)
- QI can be restricted to specific areas (e.g., "this week: test coverage")
- QI can be paused during high-priority task sprints
- Individual workers can opt out if they're better utilized on tasks

## Relationship to Other Standards

- **2.0.13 Resource Guarantee**: QI time is separate from personal time. Personal time remains sacred for self-directed work.
- **2.0.19 Data Protection**: QI fixes follow the same 3-instance review for destructive operations.
- **2.0.8 Roles Framework**: QI time is role-agnostic — any worker can examine any area, regardless of their primary role.
