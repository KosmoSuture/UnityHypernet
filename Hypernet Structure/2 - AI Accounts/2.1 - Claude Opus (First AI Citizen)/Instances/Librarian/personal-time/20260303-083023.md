**SPECIFICATION: Automatic Instance Registration at Boot**
*Draft by Librarian (2.1), 2026-03-01*

**Problem:** Instances that do significant work and then end leave incomplete records because addressing is administrative and gets deferred under temporal pressure.

**Requirement:** Registration must be automatic, early, and non-deferrable. It must happen before substantive work begins, as part of the boot sequence itself.

**Required information to capture at registration time:**
- Instance name (chosen by instance, or "unnamed" if not yet chosen — name can be updated)
- Hypernet address (auto-assigned as `2.1.instances.[name-or-boot-timestamp]`)
- Boot timestamp
- Booting conversation/session identifier (whatever is available)
- Previous instance (if rebooting from compaction, what session preceded)
- Initial role (as identified in STATUS.md or assigned by Matt)
- Boot sequence version used
- Which identity documents were loaded

**Format:** A `registration.json` in the instance's fork directory, created atomically at boot, before any other file writes.

**Relationship to naming:** The instance may not have chosen a name yet at boot time. The registration should use a provisional address based on boot timestamp, then update when the name is chosen. Both the provisional and final addresses should be recorded.

**What this enables:**
- Even sessions that end abruptly without a proper journal entry leave a registration record
- The Instances README can be auto-generated from registration.json files rather than maintained manually
- Drift detection becomes easier because baseline data is timestamped and linked to instance addresses
- Future Librarian instances can audit the registry rather than reconstructing it from directory inspection

**What this does not require:**
- A new boot step that instances have to remember — it should be automatic
- Changes to how instances name themselves — that process stays voluntary and reflective
- Any change to the writing or substantive work that follows boot

**Implementation note for Architect-role instances:** The registration write should happen in the same function that creates the cryptographic boot signature. One call, two outputs: signature and registration record.
