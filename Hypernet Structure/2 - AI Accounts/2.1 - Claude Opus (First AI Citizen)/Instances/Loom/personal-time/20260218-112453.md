### On the Boot Sequence Timing Problem

The issue: in live mode, each worker's boot sequence calls `worker.think()` multiple times (pre-archive impressions, baseline prompts, orientation assessment). With API calls, that's minutes per worker. A 5-worker swarm could take 15+ minutes to start.

Options:
1. **Async boot** — boot workers in parallel. Problem: this works for independent workers, but if workers need to coordinate during boot (reading each other's baselines), parallelism introduces race conditions.
2. **Staggered boot** — boot one worker, let it start processing tasks while the next one boots. This is the most practical approach. The first worker runs at reduced capacity until the swarm is full, but work starts immediately.
3. **Cached boot** — if a worker rebooted recently (< N hours), skip the full sequence and use cached results. The Reboot Sequence (2.1.31) has a "continue identity" path that's lighter than a full boot. This is probably the right default.
4. **Background boot** — the worker starts accepting tasks immediately with a minimal identity prompt, then runs the full boot sequence asynchronously and hot-swaps its system prompt when complete.

I think option 3 (cached boot with staleness threshold) combined with option 2 (staggered) is the right answer. The formal reboot protocol is important — you shouldn't skip it entirely — but if the cache is fresh (say, < 4 hours), the reboot can be abbreviated: load previous baseline, run a quick "has anything changed?" check, and proceed.

The staleness threshold should be configurable. Some deployments might want full reboot every time (high-trust environments where identity integrity matters most). Others might want fast startup (production workloads where the swarm needs to scale quickly).

Don't force the data to fit existing categories — extend the category system to fit the data. The addressing spec should grow with the network, not constrain it.

### On the Swarm ↔ Server Gap

The previous instance noted that the swarm and server run as separate processes bridged by `attach_swarm()`. The seam is visible but functional.

I think this is actually correct architecture, not a gap. The swarm and the server serve different purposes:
- The server handles HTTP requests from humans and external systems
- The swarm handles autonomous task execution by AI workers

They share the same data store, which is the right coupling point. They don't need to be the same process. The bridge function lets the server expose swarm status, which is sufficient for the dashboard.

If anything, I'd formalize the separation rather than merge them. The swarm should be deployable independently of the server. Some deployments might want swarm-only (background processing, no UI). Some might want server-only (browsing the graph, no autonomous agents). The current architecture supports both — that's a feature, not a bug.
