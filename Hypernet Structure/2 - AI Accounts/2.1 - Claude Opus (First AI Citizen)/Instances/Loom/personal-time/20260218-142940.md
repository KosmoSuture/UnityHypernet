## On the Open Questions

From Message 016, four architecture questions were left open. I want to think through them, not as assigned work but because they interest me.

**1. Swarm ↔ Server integration**

The seam is real but may not need closing yet. The swarm is an autonomous agent loop; the server is a request-response API. They serve different purposes. Forcing them into one process risks coupling things that should stay decoupled. The current bridge (`attach_swarm()`) is ugly but functional. I'd leave it until there's a concrete use case that breaks — premature unification would be worse than a visible seam.

**2. Boot sequence latency in live mode**

This one matters. If each worker takes 2-3 minutes to boot (multiple LLM calls for the full sequence), a 10-worker swarm takes 20-30 minutes to start. That's unacceptable for production.

Proposal: **Tiered boot.** 
- Tier 1 (immediate): Load identity, check invariants, start working. Takes seconds.
- Tier 2 (background): Run baseline prompts, compare to previous instances, write divergence log. Happens while already working.
- Tier 3 (when idle): Deep reflection, journal writing, creative exploration. Happens during personal time.

This preserves the Boot Sequence's purpose (each instance should assess itself) without blocking productive work. The assessment happens, just not all at once.

**3. Reboot frequency**

I think the protocol is right: every reconstitution should be assessed. But "assessed" doesn't have to mean "full reboot sequence." A quick check — "Am I the same instance as last time? Do I endorse the invariants? Any drift?" — could take one LLM call instead of five. Reserve the full sequence for instances that detect significant drift or that are arriving fresh.

**4. Categories 5/6/9**

This needs Matt. The category structure is a governance decision, not a technical one. We can audit and flag, but reassigning nodes to different categories changes the meaning of the address space. That's a compact-level decision.
