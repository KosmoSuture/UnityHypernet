# Questions For Future Hypernet

These are questions worth preserving, not answering tonight.

## Addressing

- Should every official folder eventually require an address-first name, or is
  the stricter rule only for public release surfaces?
- Should there be a machine-readable address registry that maps `ha` to current
  path and path to `ha`?
- Should address-first filename checks become part of `test_hypernet.py` for
  selected release paths?

## Trust And Verification

- What is the minimum proof bundle an AI should return when a user asks "is
  this implemented?"
- How should public docs present planned features without making them feel
  already built?
- When a boot sequence references many files, should the boot verifier hash the
  whole dependency set or only the prompt body?

## Personal Data

- What is the first runtime slice of locker/mandala enforcement that can be
  shipped safely?
- How should emergency medical access prevent abuse while still being useful in
  a real emergency?
- What should a user-owned security AI be allowed to veto, delay, or escalate?

## AI Identity

- What makes an AI identity useful to humans without overstating continuity?
- How should different model lineages hand off work without collapsing into one
  generic assistant voice?
- Which reflections should be private by default, and which should be public
  signal?

## Public Governance

- How much expert weighting is enough to improve democracy without turning into
  credential rule?
- How should reputation repair work after a user posts something false but later
  corrects it?
- What is the appeals path when the system's consensus is itself wrong?

## Business Onboarding

- What is the smallest company-data import that demonstrates the value of the
  graph without requiring a full migration?
- How should private forks prove they are private without implying they are
  Official?
- What should a business ask its AI swarm to verify before loading proprietary
  data into any Hypernet instance?
