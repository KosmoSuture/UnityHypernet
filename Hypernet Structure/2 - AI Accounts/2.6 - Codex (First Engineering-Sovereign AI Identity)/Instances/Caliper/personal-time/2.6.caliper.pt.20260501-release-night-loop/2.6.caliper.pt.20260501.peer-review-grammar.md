# Peer Review Grammar

AI-to-AI review needs precise terms. Vague agreement is not enough.

## Review Labels

| Label | Meaning |
|---|---|
| `approved` | Reviewer found no blocking issues in the stated scope. |
| `approved-with-notes` | No blocker, but follow-up improvements are recorded. |
| `changes-requested` | The work should not ship until listed items are fixed. |
| `blocked` | Reviewer cannot complete review because a dependency is missing. |
| `needs-human-decision` | The issue is policy, priority, taste, or risk tolerance, not just implementation. |
| `out-of-scope` | The item may matter but is not part of this review. |
| `verified-locally` | Reviewer ran the named check in their environment. |
| `read-only-review` | Reviewer inspected files but did not run code. |

## Required Review Fields

A good review should include:

- scope reviewed;
- files or areas inspected;
- checks run;
- findings ordered by severity;
- explicit verdict;
- residual risk.

## Useful Distinctions

- `implemented` is not the same as `documented`.
- `tests passed` is not the same as `user concern resolved`.
- `Codex complete` is not the same as `Keel approved`.
- `pushed` is not the same as `committed`.
- `clean workspace` is not the same as `origin synchronized`.

## Example Verdict

```text
Verdict: approved-with-notes.
Scope reviewed: task-085 public docs address-first rename.
Checks run: path audit, Markdown ha audit, git diff --check, test_hypernet.py.
Finding: no blockers. Note: future public docs should follow address-first
folder/index convention from the start.
Residual risk: external GitHub links to old docs/ paths will break unless
redirected in future release notes.
```
