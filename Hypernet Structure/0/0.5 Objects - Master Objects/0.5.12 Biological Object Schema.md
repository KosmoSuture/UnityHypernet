# 0.5.12 — Biological Object Schema

## Overview

Biological objects represent living organisms, biological processes, health records, and genetic data. This category covers everything from a single cell to an entire ecosystem, from a blood test result to a complete genome sequence, from an individual organism to a species-level population record.

**Why a separate category:** Biological objects have a unique structural signature: lifecycle state, taxonomic classification, and biological properties (health status, genetic markers, ecological relationships) that no other object type has as essential properties. Forcing biological data into Document or Media loses the semantic precision needed for queries like "What species were observed in this habitat?" or "Show my lab results from the last six months." Health records, in particular, carry strict regulatory requirements (HIPAA, GDPR) that demand first-class treatment.

**Parent type:** 0.5.0 (Master Object Schema)

### 0.5.12.2 Health Record

A record of health-related information pertaining to a person or organism. All Health Record subtypes carry elevated privacy requirements.

**Privacy note:** Health records are among the most sensitive data in the Hypernet. All 0.5.12.2.x objects should default to `visibility: "private"` and be flagged `0.8.2.sensitive`. Data is subject to **HIPAA** (US), **GDPR** (EU), and equivalent regional health privacy regulations. Access control must be enforced at the object level, and audit logging is mandatory.

**Additional fields:**
```yaml
health_record:
  subject_ha: "[HA of patient/person]"  # Required — who this record is about
  provider_ha: null                     # HA of healthcare provider (0.5.1 or 0.5.2)
  facility_ha: null                     # HA of facility/hospital (0.5.2)
  encounter_date: null                  # ISO 8601
  record_type: "diagnosis | lab | prescription | procedure | vitals"
  status: "active | amended | corrected | entered-in-error | inactive"
  confidentiality: "normal | restricted | very-restricted"
  consent_ha: null                      # HA of consent document authorizing this record
  regulatory_flags:
    hipaa: false                        # Subject to HIPAA
    gdpr: false                         # Subject to GDPR
    jurisdiction: null                  # Applicable jurisdiction code
```

## Methods

### classify_organism
```yaml
method:
  name: "classify_organism"
  description: "AI identifies and classifies an organism from images, descriptions, or partial data"
  handler: "ai"
  ai_context:
    - "Input image or description of organism"
    - "Known geographic location and habitat"
    - "Taxonomic databases and field guides"
  inputs:
    - observation: biological_object     # Image, description, or partial organism record
    - location_ha: string               # Optional — where it was observed
  outputs:
    - scientific_name: string
    - common_name: string
    - taxonomy: dict                    # Full taxonomic classification
    - conservation_status: string
    - confidence: float
    - similar_species: list             # Names of commonly confused species
```

### analyze_health_trend

```yaml
method:
  name: "analyze_health_trend"
```

```yaml
    - carrier_status: list              # [{condition, carrier_type, variant}]
    - pharmacogenomic_flags: list       # [{drug, gene, effect, recommendation}]
    - summary: string                   # Natural language genetic risk summary
    - recommended_screenings: list      # Suggested follow-up genetic tests or clinical screens
```

## Privacy & Security Notes

Biological data includes some of the most sensitive information in the Hypernet:

1. **Health records (0.5.12.2.x)** must default to `visibility: "private"` and be flagged `0.8.2.sensitive`. These records are subject to **HIPAA** (US), **GDPR** (EU), and equivalent regional health privacy regulations.
2. **Genetic data (0.5.12.3.x)** carries heightened sensitivity — genetic information can reveal predispositions, ancestry, and family relationships. Flag with `0.8.2.sensitive` and apply strictest access controls. The Genetic Information Nondiscrimination Act (GINA) provides additional US protections.
3. **Prescriptions (0.5.12.2.3)** may reveal conditions and should never be visible to unauthorized parties.
4. **Vital signs (0.5.12.2.5)** from wearables and devices should default to `visibility: "private"`.
5. **Ancestry profiles (0.5.12.3.4)** can reveal ethnic background and familial connections — treat as sensitive personal data under GDPR Article 9.
6. **Compliance:** The `flags` field and `regulatory_flags` subfield should indicate all applicable compliance requirements. Audit logging (who accessed what, when) is mandatory for all 0.5.12.2.x and 0.5.12.3.x objects.
