# 0.5.18 App Load Object Schema

## Overview

An **App Load** is a structured initialization artifact for a
Hypernet-aware application. Where a Boot Sequence (`0.5.17`)
initializes an AI into a role, an App Load initializes an app into
a known runtime shape: what it is allowed to read, what it may
write, which connectors it uses, which user/company/AI accounts it
operates under, which AI helpers it may call, and how a user or
their security AI can verify those claims.

App loads are first-class Hypernet objects. They inherit the base
object model from `0.5.0`, carry the master `integrity.hash`
property, and can be checked against the Official registry before
a user, company, or AI allows the application to run.

The full schema below is a v1 design surface. Runtime installation,
permission grants, and Official registry verification are planned
implementation work.

## Schema Definition

```yaml
app_load_object:
  # BASE OBJECT (inherited from 0.5.0)
  identity:
    address: "[address]"               # e.g., 0.5.18.examples.public-alpha-tour
    object_id: "[UUID]"
    object_type: "app_load"
    subtype: "web|desktop|mobile|service|connector|agent-tool|vr-gis"
    version: "1.0.0"

  # INTEGRITY (master hash from 0.5.0)
  integrity:
    hash:
      algorithm: "sha256"
      value: "[hex digest of canonicalized app-load manifest]"
      computed_at: "2026-05-01T12:00:00Z"
      by: "[Mandala or AI instance that verified]"
      scope: "content"
    verified: true
    verification_chain:
      - verifier: "[Mandala ID]"
        verified_at: "2026-05-01T12:00:00Z"
        method: "official-attestation"
        result: "passed"

  content:
    # APP IDENTITY - what is being loaded
    app:
      name: "[human-readable app name]"
      app_address: "[canonical app object or repository address]"
      publisher: "[person/company/AI account address]"
      repository: "[git URL or Hypernet source object]"
      release_channel: "official|private-fork|dev|test"
      runtime_kind: "browser|native|server|worker|agent|vr"

    # ENTRYPOINTS - where execution begins
    entrypoints:
      primary:
        kind: "http|cli|python|node|container|static|vr-scene|android|wear-os|native"
        target: "[path, URL, command, container image, or scene]"
        args: []
      secondary:
        - name: "[optional named entrypoint]"
          kind: "[kind]"
          target: "[target]"

    # DEPENDENCIES - code, data, services, models
    dependencies:
      code:
        - address: "[source-code object or file path]"
          required_hash: "[sha256]"
      data:
        - address: "[Hypernet address/prefix]"
          access_mode: "read|write|admin"
      services:
        - name: "[service name]"
          endpoint: "[URL or local socket]"
          mode: "official|private-fork|local|external-oauth|third-party-api"
      models:
        - role: "assistant|classifier|security-sentry|planner"
          model_policy: "user-choice|local-first|publisher-recommended"

    # PERMISSIONS - minimum privileges this app requests
    permissions:
      requested_scopes:
        - scope: "[address prefix or named capability]"
          access: "read|write|delete|share|execute"
          reason: "[plain-English reason shown to user/security AI]"
          required: true
      denied_by_default:
        - "1.* master-account direct access"
        - "unbounded private locker scan"
        - "write outside declared prefixes"
      grant_surface: "mandala|company-role|ai-boot-attestation|iot-device|user-confirmation|security-ai-review|oauth"

    # DATA BINDINGS - what objects the app reads/writes
    data_bindings:
      reads:
        - address_pattern: "[prefix or query]"
          object_types: ["0.5.x"]
          link_types: ["0.6.x"]
      writes:
        - address_pattern: "[prefix]"
          object_types: ["0.5.x"]
          validation: "schema-first|staged-write|required-review"
      link_policy:
        source_write_required: true
        target_read_required: true
        sensitive_links_need_consent: true

    # AI HELPERS - optional AI processes the app may invoke
    ai_helpers:
      allowed: true
      boot_sequences:
        - address: "[0.5.17 boot sequence address]"
          purpose: "[why this helper is loaded]"
          max_visibility: "public|group|private"
      process_loads:
        - path: "[process-load markdown path]"
          trigger: "[when to load it]"
      model_adapters:
        - name: "[adapter name]"
          phase: "deterministic-local|provider-backed|hybrid"
          provider: "local|openai|anthropic|google|self-hosted|other"
          model: "[model or deterministic adapter id]"
          purpose: "[why this adapter is used]"
          requires_private_secrets: false
          audit_metadata:
            - "provider"
            - "model"
            - "adapter"
            - "response_source"
      security_review:
        required_for_private_data: true
        security_ai_address: "[user-owned security AI, optional]"

    # USER INTERFACE AND SURFACES
    surfaces:
      primary_surface: "web|desktop|mobile|api|vr|gis|chat"
      public_routes: []
      authenticated_routes: []
      admin_routes: []
      accessibility_requirements:
        - "plain-language permission summary"
        - "visible Official/Private mode indicator"

    # OBSERVABILITY - what is logged and where
    observability:
      audit_log_address: "[where app events are written]"
      records:
        - "loads"
        - "permission prompts"
        - "data reads"
        - "data writes"
        - "AI helper calls"
        - "errors"
      user_exportable: true

    # ROLLBACK AND REVOCATION
    lifecycle:
      install_state: "draft|reviewed|official-attested|published|revoked"
      rollback_strategy: "none|snapshot|journal-replay|migration-down"
      revoke_effect:
        - "stop new launches"
        - "warn active sessions"
        - "preserve audit logs"

  methods:
    verify:
      input: "app_load_address"
      output: "verification_result"

    resolve_dependencies:
      input: "app_load_address"
      output: "dependency_report"

    authorize:
      input: "app_load_address, actor_address, requested_scopes"
      output: "mandala_grant_request"

    install:
      input: "app_load_address, target_account"
      output: "installation_record"

    launch:
      input: "installation_record, runtime_context"
      output: "session_address"

    audit:
      input: "app_load_address, audit_scope"
      output: "audit_report"

    revoke:
      input: "app_load_address, reason"
      output: "revocation_record"

  links:
    - link_type: "loads_application"
      target: "[application/source-code object]"
    - link_type: "governed_by"
      target: "[privacy/security/access standard]"
    - link_type: "uses_boot_sequence"
      target: "[0.5.17 boot sequence address]"
    - link_type: "requests_access_to"
      target: "[address prefix, locker, or company role]"
    - link_type: "verified_by"
      target: "[Official registry attestation address]"
```

## Required Fields

An app load object MUST have:

- `identity.address`
- `identity.object_id`
- `identity.object_type` == `"app_load"`
- `content.app.name`
- `content.app.publisher`
- `content.entrypoints.primary`
- `content.permissions.requested_scopes`
- `content.data_bindings`
- `content.observability.audit_log_address`

An app load published to the **Official registry** MUST
additionally have:

- `integrity.hash.value` computed over the canonicalized app-load
  manifest
- `integrity.verified == true`
- `integrity.verification_chain` with at least one
  `method: "official-attestation"` entry
- a visible Official/Private mode indicator in the user-facing
  surface or API response

## Rules for Creating App Loads

1. **Declare the runtime honestly.** A web app, local service,
   connector, VR scene, and AI tool have different risks. The
   subtype and entrypoint must match reality.
2. **Ask for minimum permissions.** Every requested scope carries
   a plain-English reason. Broad account or locker access is
   rejected unless the app can justify it.
3. **No hidden writes.** All write prefixes and link-writing rules
   must appear in `data_bindings.writes`.
4. **Private data requires a security review path.** If the app
   touches personal lockers, company private data, IoT device
   state, or AI identity archives, the load must declare how a
   security AI or equivalent reviewer can inspect the request.
5. **AI helpers are explicit.** Apps may use AI assistants, but
   each helper references a `0.5.17` boot sequence or equivalent
   identity record and explains why it is needed.
6. **Deterministic adapters come before provider secrets.** Phase
   0.5 app loads should declare a deterministic local adapter
   first when possible, so clients and tests can build against
   stable behavior without private provider credentials. Live
   provider adapters should arrive in later app-load revisions or
   be explicitly marked `provider-backed` with provider/model audit
   metadata.
7. **Official/Private mode is visible.** The user and their AI
   must be able to tell whether the app is running from an
   Official attested load or a Private fork.
8. **Hash is computed last.** The app-load manifest is finalized,
   canonicalized, hashed, and then attested. Any in-scope change
   invalidates the prior hash.
9. **Audit records are part of the load.** If an app cannot say
   where load events, permission prompts, reads, writes, and AI
   helper calls are logged, it is not ready for Official use.

## Hash Computation Specifics

For app loads, the canonical hashable content is the complete
`content` section after these normalizations:

- UTF-8, no BOM
- YAML/JSON keys sorted recursively
- Line endings normalized to LF
- No trailing whitespace on lines
- Empty optional arrays normalized to `[]`
- Dynamic fields such as current timestamp, temporary port, or
  local install path excluded unless explicitly declared in scope

This lets a user's AI answer: "Is this the exact app load the
Official registry attested, or has something changed?"

| `mobile` | Phone/tablet app | NFC emergency medical locker access |
| `service` | Backend service | Embedded graph index service |
| `connector` | Data migration/integration | Gmail/import connector |
| `agent-tool` | Tool invoked by an AI helper | Local file scanner |
| `vr-gis` | Spatial/3D application | VR/GIS graph viewer |

## App Load Lifecycle

```
draft -> reviewed -> official-attested -> published -> revoked
```

- **draft**: manifest being written, hash not final.
- **reviewed**: schema and permissions reviewed, hash computed,
  not yet Official.
- **official-attested**: registered on the Official registry.
- **published**: usable by users, companies, AI assistants, or
  IoT devices depending on declared scope.
- **revoked**: no longer canonical; new launches should stop,
  active sessions should warn, audit logs remain preserved.

## Relationship to Boot Sequences

Boot sequences (`0.5.17`) initialize AI identity. App loads
(`0.5.18`) initialize application behavior. They can be composed:
an app load can call one or more AI helpers, and each AI helper
can be booted by a `0.5.17` boot sequence.

The boundary matters:

- A boot sequence says "who the AI is while acting."
- An app load says "what this application is allowed to do."
- A personal security AI can inspect an app load before allowing
  any mandala grant.

For app loads that invoke model-backed behavior, a deterministic
local adapter is the recommended Phase 0.5 default. It is not the
final assistant brain; it is a test/dev contract that lets clients
build, verify scopes, and pass repeatable tests before any private
provider key exists. A provider-backed adapter should declare its
provider, model, credential boundary, and response-source audit
metadata before it is considered for Official use.
