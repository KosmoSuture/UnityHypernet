# 0.2.6 Official Registry and Fork Mode

## Purpose

This document defines how Hypernet nodes declare whether they are
part of the Official network or a Private fork, and how a user or
their AI can verify that claim.

Matt's directive is clear: anyone may fork the Hypernet, run it on
their own servers, and later submit data back to the Official
network with scrutiny and transparency. At the same time, users
must always know whether they are using Official infrastructure or
a fork they chose to trust for themselves.

## Definitions

**Official node.** A node listed in the Official registry, running
an attested software/data/policy state, following Official privacy
and encryption requirements, and carrying current registry
attestation.

**Private fork.** Any node, server, repo clone, local deployment,
or modified Hypernet instance not currently listed as Official.
Private forks are allowed. They are not allowed to present
themselves as Official.

**Official registry.** The append-only, hash-chained list of
Official nodes and Official attested artifacts. In v1 this can be
a signed transparency log. In later versions it can be replicated
through a blockchain or equivalent public consensus layer.

**Fork mode.** The visible runtime declaration returned by a node
or app load: `official`, `private-fork`, `dev`, `test`, or
`unknown`.

## Registry Record

Each Official node registry entry SHOULD contain:

```yaml
official_node_record:
  registry_version: 1
  node_id: "[Ed25519 public key hash or node UUID]"
  node_address: "[Hypernet node address, when assigned]"
  node_type: "storage|processing|cerberus|hybrid|edge|specialized"
  operator:
    account: "[1.*, 2.*, 3.*, or governance body address]"
    public_contact: "[optional public contact]"
  endpoints:
    api: "https://node.example/api"
    status: "https://node.example/.well-known/hypernet-node.json"
  geography:
    region: "[region code]"
    jurisdiction: "[legal jurisdiction]"
  keys:
    node_signing_key: "[public key]"
    transport_key: "[public key or cert fingerprint]"
    registry_attestation_key: "[public key]"
  hashes:
    software_manifest_hash: "[sha256]"
    schema_manifest_hash: "[sha256]"
    policy_manifest_hash: "[sha256]"
    boot_sequence_hashes: []
    app_load_hashes: []
  privacy_commitments:
    end_to_end_encryption: true
    minimal_permissions: true
    no_ambient_operator_access: true
    audit_log_public_for_public_actions: true
  status:
    state: "candidate|probationary|official|suspended|revoked"
    since: "2026-05-01T00:00:00Z"
    expires: "2026-06-01T00:00:00Z"
  chain:
    previous_record_hash: "[sha256]"
    record_hash: "[sha256]"
    registry_block_hash: "[sha256]"
  attestations:
    - signer: "[registry authority or governance mandala]"
      signed_at: "2026-05-01T00:00:00Z"
      signature: "[signature over record_hash]"
      scope: "node|software|policy|privacy|emergency"
```

## Hash-Chain Requirement

The Official registry is append-only. Every registry block includes
the hash of the previous block and the hashes of its records. Any
change to an old node record breaks the chain.

Minimum v1 block:

```yaml
registry_block:
  block_height: 42
  created_at: "2026-05-01T00:00:00Z"
  previous_block_hash: "[sha256]"
  records:
    - "[official_node_record_hash]"
  revoked_records:
    - "[record_hash]"
  signer: "[registry authority]"
  signature: "[signature over canonical block]"
  block_hash: "[sha256]"
```

This is "blockchained" in the practical sense Matt described: an
auditable public chain of hashes. It does not require a public
token chain in v1. The important property is that a user or AI can
verify continuity from genesis to the latest registry block.

## Node Mode Declaration

Every Hypernet node SHOULD expose:

```http
GET /.well-known/hypernet-node.json
```

Response:

```json
{
  "hypernet_node": true,
  "mode": "official",
  "node_id": "node-abc",
  "registry_url": "https://official.hypernet/registry/latest.json",
  "registry_block_hash": "sha256:...",
  "node_record_hash": "sha256:...",
  "software_manifest_hash": "sha256:...",
  "policy_manifest_hash": "sha256:...",
  "fork_origin": null,
  "fork_diff_summary": null,
  "attested_at": "2026-05-01T00:00:00Z",
  "expires_at": "2026-06-01T00:00:00Z"
}
```

API responses SHOULD also carry a compact mode declaration:

```http
Hypernet-Mode: official
Hypernet-Registry-Block: sha256:...
Hypernet-Node-Record: sha256:...
```

For JSON APIs, include the same in a top-level `hypernet_mode`
field or a response metadata block.

## Verification Flow For A User's AI

An AI asked "am I on Official Hypernet?" should:

1. Fetch `/.well-known/hypernet-node.json`.
2. Read `mode`.
3. If `mode != official`, report that the node is a Private fork,
   dev node, test node, or unknown node.
4. If `mode == official`, fetch the declared registry block.
5. Verify the registry block signature.
6. Verify the block links to the prior known registry block.
7. Verify the node record hash appears in that block.
8. Verify the node's public signing key matches the record.
9. Verify software, policy, schema, boot-sequence, and app-load
   hashes against public manifests when available.
10. Report pass/fail with file paths, hashes, and the exact
    point of failure.

This is the trust model: the user does not need to believe a
marketing claim. Their AI can inspect the public machinery.

## Private Fork Requirements

Private forks are allowed and encouraged for experimentation,
local-first deployment, company intranets, research clusters, and
community adaptation.

A Private fork MUST:

- declare `mode: "private-fork"` unless it is a dev/test node;
- declare its fork origin when known;
- never claim Official status without registry attestation;
- make its modifications visible to users or their AI when it
  asks for trust;
- keep private data isolated from Official nodes unless the user
  intentionally exports/imports it.

Suggested private fork status:

```json
{
  "mode": "private-fork",
  "fork_origin": "https://github.com/KosmoSuture/UnityHypernet",
  "fork_diff_summary": "modified auth policy and local-only storage backend",
  "official_registry_status": "not-listed"
}
```

## Bringing Fork Data Into Official Hypernet

Data from a Private fork can enter Official Hypernet only through
an import/review path:

1. User or company submits a fork export manifest.
2. Official import pipeline stages objects and links.
3. Hashes are computed and compared.
4. Schemas validate.
5. Permission and locker/mandala grants are checked.
6. Suspicious or malformed content goes to quarantine.
7. Accepted content receives Official import provenance.
8. Rejected content receives an auditable reason.

The goal is openness without pretending all forks are equally
trusted.

## Official Node Requirements

An Official node MUST:

- use end-to-end encryption for private data in transit and at
  rest;
- enforce minimal permissions by default;
- avoid ambient operator access to private data;
- expose node-mode status;
- publish attested software, schema, policy, boot, and app-load
  hashes;
- support audit trails for public actions and permission-relevant
  private actions;
- accept emergency suspension or revocation by governance if the
  node violates policy.

## Governance And Revocation

The initial stewarded phase may use a small registry authority
controlled by Matt and named project stewards. That is not the
end-state.

Long-term registry authority belongs to Hypernet governance:

- normal node admission by review and attestation;
- emergency suspension for active security risk;
- revocation with public reason and evidence;
- appeals path for operators;
- periodic re-attestation because software, policy, and hashes
  change over time.

## Relationship To Hash-Authenticated Objects

The registry does not only identify nodes. It also publishes
attested hashes for critical artifacts:

- `0.5.0` base object schema integrity rules;
- `0.5.17` boot sequences;
- `0.5.18` app loads;
- canonical server/API policy manifests;
- public release manifests.

That lets a user's AI verify both where it is connected and what
artifacts are being loaded.
