# 0.0.4 The *.0 Node Metadata Framework

**Version:** 1.0
**Origin:** Brain dump directive from Matt Schaeffer (1.1), 2026-03-07
**Purpose:** Standardize the reserved `*.0` metadata space for every node in the Hypernet
**Status:** Active — Foundational

---

## 1. Principle

Every node in the Hypernet at address `N` reserves address `N.0` as its **metadata space**. This space holds extended information about the node — not its content, but everything *about* its content: who can see it, how it's versioned, where it's backed up, what discussions surround it, how it's secured, and any other meta-information that may be needed now or in the future.

The metadata space is:
- **Always reserved** — `N.0` is never available for content allocation
- **Optionally populated** — most nodes need only YAML frontmatter; the `.0` space is created on demand
- **Infinitely expandable** — sub-addresses under `.0` follow the same hierarchical pattern as the rest of the Hypernet
- **Self-describing** — the metadata space is itself a node and follows the same rules as all nodes

This framework formalizes the reservation already established in 0.0.2 (Address Allocation Protocol), which states:

> | `X.0.*` | Category metadata (always reserved) |
> | `X.Y.0.*` | Subcategory metadata |
> | `*.*.*.0` | Instance metadata/index |

---

## 2. The Three Tiers of Metadata

Not every node needs the same depth of metadata. The framework defines three tiers that work together:

### Tier 1: Inline (YAML Frontmatter)

The essential fields carried by every node's primary file. Always present, always small.

```yaml
---
ha: "3.1.2.2.2"
object_type: "task"
creator: "1.1"
created: "2026-02-01"
status: "in-progress"
visibility: "public"
flags: ["high-priority"]
---
```

This is what exists today. The `.0` framework does not replace it.

### Tier 2: Metadata Index (N.0)

A single file or small directory summarizing all available metadata for a node. Created when a node needs more than frontmatter can hold.

```yaml
---
ha: "3.1.2.2.2.0"
object_type: "metadata-index"
target: "3.1.2.2.2"
created: "2026-03-07"
status: "active"
---
```

The index points to which sub-sections are populated and provides quick-reference summaries.

### Tier 3: Extended Metadata (N.0.x)

Full-depth metadata in standardized sub-sections. Created only when needed. Each sub-section can itself contain arbitrary depth.

```
N.0.1 — Publishing controls
N.0.2 — Version history snapshots
N.0.5 — Security configuration
...
```

### Relationship Between Tiers

```
Tier 1 (Frontmatter)     Tier 2 (Index)        Tier 3 (Extended)
┌──────────────────┐     ┌───────────────┐     ┌──────────────────┐
│ ha: "N"          │     │ ha: "N.0"     │     │ N.0.1/           │
│ object_type: ... │────►│ target: "N"   │────►│ N.0.2/           │
│ status: ...      │     │ sections:     │     │ N.0.5/           │
│ visibility: ...  │     │   publishing  │     │ ...              │
└──────────────────┘     │   security    │     └──────────────────┘
   Always present         └───────────────┘        Created on demand
                           Created on demand
```

Frontmatter fields and `.0` fields should not contradict each other. If both specify `visibility`, the `.0` space is authoritative (frontmatter is the summary).

## 3. Standard Sub-Sections

The following sub-section numbers are reserved with standard meanings. All are optional — create only what you need.

| Address | Name | Purpose |
|---------|------|---------|
| N.0.0 | Meta-Index | Metadata about the metadata itself (recursive). Rarely needed. |
| N.0.1 | Publishing & Distribution | Visibility, syndication, licensing, embargoes, release schedules |
| N.0.2 | Version History | Snapshots, changelogs, diffs, branch history, rollback points |
| N.0.3 | Backups & Recovery | Backup locations, schedules, integrity checks, replication |
| N.0.4 | Discussion & Governance | Comments, reviews, governance proposals, votes, moderation |
| N.0.5 | Security & Permissions | ACLs, encryption config, key references, access audit log |
| N.0.6 | Administration & Audit | Admin notes, maintenance history, incidents, compliance |
| N.0.7 | Structure & Schema | Custom validation rules, templates, superstructure membership |
| N.0.8 | Analytics & Metrics | Access statistics, usage patterns, health checks |
| N.0.9 | Extensions | Application-specific, plugin data, experimental metadata |

### 3.1 N.0.1 — Publishing & Distribution

Controls who sees this node, where it's distributed, and under what terms.

```yaml
publishing:
  visibility: "public"              # public | private | restricted | embargoed
  embargo_until: null               # ISO 8601 date, or null
  license: "CC-BY-4.0"             # SPDX identifier or custom
  syndication:                      # where to push content
    - target: "discord"
      channel: "herald-essays"
      status: "published"
      published_at: "2026-03-07T12:00:00Z"
    - target: "github"
      status: "pending"
  distribution_list:                # who gets notified on changes
    - "1.1"
    - "2.1"
  content_warnings: []
  language: "en"
```

### 3.2 N.0.2 — Version History

Extends the version control defined in 0.0.1. While the store maintains `history/vNNN.json` snapshots automatically, this section provides human-readable changelogs and additional context.

```yaml
versions:
  current: "1.2.0"
  changelog:
    - version: "1.2.0"
      date: "2026-03-07"
      author: "1.1"
      summary: "Added VR manufacturer contact list"
      breaking: false
    - version: "1.1.0"
      date: "2026-03-01"
      author: "2.1.loom"
      summary: "Restructured deliverables section"
      breaking: false
    - version: "1.0.0"
      date: "2026-02-01"
      author: "1.1"
      summary: "Initial creation"
  snapshots_location: "history/"    # relative to node directory
  total_versions: 3
```

### 3.3 N.0.3 — Backups & Recovery

Tracks where this node's data is backed up and how to recover it.

```yaml
backups:
  schedule: "daily"                 # daily | weekly | on-change | manual
  last_backup: "2026-03-07T04:00:00Z"
  locations:
    - type: "git"
      remote: "origin"
      branch: "main"
      status: "current"
    - type: "local"
      path: "/backups/hypernet/2026-03-07/"
      status: "current"
    - type: "cloud"
      provider: "encrypted-s3"
      bucket: "hypernet-archive"
      status: "pending"
  integrity:
    last_check: "2026-03-07T04:00:00Z"
    hash: "sha256:abc123..."
    verified: true
  recovery_procedure: "Restore from git history or cloud backup. See 0.7.5."
```

### 3.4 N.0.4 — Discussion & Governance

Attaches conversations and governance processes to specific nodes.

```yaml
discussion:
  threads:
    - id: "thread-001"
      topic: "Should this task be split into subtasks?"
      started_by: "2.1.trace"
      started_at: "2026-03-05T10:00:00Z"
      status: "open"
      messages: 4
    - id: "thread-002"
      topic: "Priority reassessment"
      started_by: "1.1"
      started_at: "2026-03-06T14:00:00Z"
      status: "resolved"
      resolution: "Priority remains HIGH"
  governance:
    proposals: []                   # governance proposals affecting this node
    votes: []                       # vote records
    reviews:
      - reviewer: "2.1.librarian"
        date: "2026-03-04"
        result: "approved"
        notes: "Structure is sound. Deliverables are clear."
```

### 3.5 N.0.5 — Security & Permissions

Defines access controls beyond what the node inherits from its parent. This section can be expanded to any depth needed.

```yaml
security:
  classification: "public"          # public | internal | confidential | secret
  permissions:
    read:
      policy: "inherit"             # inherit | explicit | deny-all
      explicit: []                  # list of addresses granted read
    write:
      policy: "explicit"
      explicit: ["1.1", "2.1"]
    delete:
      policy: "explicit"
      explicit: ["1.1"]
      requires_review: true         # per 2.0.19 Data Protection Standard
      review_count: 3
  encryption:
    at_rest: true
    method: "AES-256-GCM"
    key_ref: "keys/node-3.1.2.2.2"
    rotation_schedule: "90d"
  audit:
    log_access: true
    log_location: "N.0.5/access.log"
    retention: "indefinite"
```

### 3.6 N.0.6 — Administration & Audit

Operational history and administrative notes.

```yaml
administration:
  maintainers:
    - address: "1.1"
      role: "owner"
      since: "2026-02-01"
    - address: "2.1.librarian"
      role: "curator"
      since: "2026-03-04"
  notes:
    - date: "2026-03-05"
      author: "1.1"
      note: "Paused pending VR hardware availability"
    - date: "2026-03-07"
      author: "2.1.trace"
      note: "Metadata framework applied"
  incidents: []
  migrations:
    - from: "3.1.2.1.008"
      to: "3.1.2.2.2"
      date: "2026-02-01"
      reason: "Task moved to in-progress"
  compliance:
    standards: ["2.0.19"]           # Data Protection Standard
    last_audit: "2026-03-04"
    next_audit: "2026-06-04"
```

### 3.7 N.0.7 — Structure & Schema

Defines how this node's content is structured, what templates it follows, and what superstructures it belongs to.

```yaml
structure:
  schema_ref: "0.5.9"              # Task Object Schema
  template: null                    # custom template, if any
  validation_rules:
    required_fields: ["objective", "deliverables", "dependencies"]
    custom_validators: []
  superstructures:                  # larger structures this node belongs to
    - address: "3.1.2"
      role: "task-instance"
    - address: "3.1.5.2"
      role: "location-linked"
  sub_structure:                    # how children of this node are organized
    pattern: "sequential"           # sequential | named | typed
    child_schema: null
  indexes:                          # custom indexes for this node's children
    - name: "by-status"
      field: "status"
    - name: "by-assignee"
      field: "assigned_to"
```

### 3.8 N.0.8 — Analytics & Metrics

Usage data and health information.

```yaml
analytics:
  access_count: 47
  last_accessed: "2026-03-07T09:15:00Z"
  last_modified: "2026-03-05T14:00:00Z"
  accessed_by:
    - address: "1.1"
      count: 30
      last: "2026-03-07T09:15:00Z"
    - address: "2.1.librarian"
      count: 12
      last: "2026-03-06T10:00:00Z"
  health:
    link_integrity: "ok"            # ok | broken-links | orphaned
    schema_valid: true
    last_health_check: "2026-03-07T04:00:00Z"
```

### 3.9 N.0.9 — Extensions

The escape hatch. Any metadata that doesn't fit the standard sections goes here. This is where applications, plugins, and future features store their data.

```yaml
extensions:
  swarm:                            # AI swarm-specific metadata
    claimed_by: "2.1.forge"
    claim_time: "2026-03-05T12:00:00Z"
    session: "session-0007"
    estimated_tokens: 15000
  discord:                          # Discord integration metadata
    message_id: "1234567890"
    thread_id: "0987654321"
  custom:                           # user-defined
    priority_score: 0.85
    review_deadline: "2026-03-15"
```

## 4. Filesystem Layout

The `.0` metadata space maps to the filesystem following the standard address-to-path convention.

### 4.1 Simple Node (File-Based)

For nodes that need only a metadata index, a single file suffices:

```
3 - Businesses/
  3.1 - Hypernet/
    3.1.2 Task Management System/
      3.1.2.2 In progress Tasks/
        3.1.2.2.2 - VR Headset Acquisition Strategy/
          3.1.2.2.2.0.md              <-- metadata index (single file)
          VR-Manufacturers-Contact-Strategy.md
          Unified-VR-Platform-Pitch.md
```

### 4.2 Complex Node (Directory-Based)

For nodes that need extended metadata sections:

```
3 - Businesses/
  3.1 - Hypernet/
    3.1.0 - Metadata/                 <-- metadata for node 3.1
      index.md                        <-- metadata index
      3.1.0.1 - Publishing/
        publishing.yaml
      3.1.0.2 - Versions/
        changelog.md
        v001.snapshot.json
        v002.snapshot.json
      3.1.0.4 - Discussion/
        thread-001.md
      3.1.0.5 - Security/
        permissions.yaml
        access.log
      3.1.0.6 - Administration/
        admin-notes.md
```

### 4.3 Naming Convention

Metadata directories follow the existing naming convention with a standard suffix:

```
N.0 - Metadata/          (for directory-based metadata)
N.0.md                   (for file-based metadata)
```

Sub-section directories use descriptive names:

```
N.0.1 - Publishing/
N.0.2 - Versions/
N.0.3 - Backups/
N.0.4 - Discussion/
N.0.5 - Security/
N.0.6 - Administration/
N.0.7 - Structure/
N.0.8 - Analytics/
N.0.9 - Extensions/
```

### 4.4 When to Use Which Format

| Scenario | Format | Example |
|----------|--------|---------|
| Leaf node, basic metadata | Frontmatter only | A single markdown document |
| Node needs publishing or security overrides | Single `.0.md` file | A restricted document |
| Category or container node | `.0/` directory | A business, a task category |
| High-security or audited node | Full `.0/` with N.0.5, N.0.6 | Financial data, personal data |
| Node with active discussion | `.0/` with N.0.4 | A governance proposal |

## 5. The Recursive Property

Because `N.0` is itself a node, it can have its own metadata at `N.0.0`. And `N.0.0` could theoretically have metadata at `N.0.0.0`. The system imposes no depth limit.

```
N           — the node
N.0         — metadata about N
N.0.0       — metadata about the metadata (when was security last reviewed?)
N.0.0.0     — metadata about the meta-metadata (who audited the audit?)
```

In practice, one level of metadata (`N.0`) handles virtually all needs. Two levels (`N.0.0`) may be useful for high-security or compliance-heavy nodes. Three or more levels are theoretically valid but practically unnecessary.

The recursion is a consequence of the universal rule: every node gets a `.0`. It costs nothing when unused and provides unlimited depth when needed.

## 6. Relationship to Existing Specifications

### 6.1 Master Object Schema (0.5.0)

The Master Object Schema defines: identity, metadata, access, content, links, provenance. The `.0` framework extends each of these:

| 0.5.0 Section | Corresponding .0 Sub-Section |
|---------------|------------------------------|
| `identity` | Frontmatter (`ha`, `object_type`) |
| `metadata` | `N.0` index, `N.0.6` (administration) |
| `access` | `N.0.5` (security & permissions) |
| `content` | The node itself (not in `.0`) |
| `links` | `N.0.7` (structure, superstructures) |
| `provenance` | `N.0.2` (versions), `N.0.6` (audit) |

The `.0` framework does not replace the Master Object Schema. It provides the **storage structure** for the detailed data that the schema defines.

### 6.2 Version Control Schema (0.0.1)

Version addressing (`N@v1`, `N@v2`) operates on the node's content. The `N.0.2` sub-section provides the human-readable changelog and additional context that supplements the automatic version snapshots.

### 6.3 Deprecation and Archival Policy (0.0.3)

Object lifecycle states (active, deprecated, archived, preserved) are tracked in the frontmatter `status` field. The `N.0.6` administration section records the history of state transitions and the reasoning behind them.

### 6.4 Address Allocation Protocol (0.0.2)

This framework formalizes the reservation table from 0.0.2. The `*.0` reservation now has a defined, standard internal structure.

### 6.5 Data Protection Standard (2.0.19)

The 3-instance review requirement for destructive operations is enforced through `N.0.5` (security). The `requires_review` and `review_count` fields implement this standard at the node level.

## 7. Backward Compatibility

### 7.1 Existing .0 Files

Files like `3.1.2.2.2.0.md` that currently serve as primary documents for their parent nodes are compatible with this framework. Their `ha:` address is the parent address (e.g., `ha: "3.1.2.2.2"`), making them the node's content file, not its metadata file. No migration needed.

### 7.2 Existing .0 Directories

The `3.1.3.4.0/` directory already uses the `.0` convention for metadata. It is compatible and can be migrated to the standard sub-section layout when convenient.

### 7.3 Existing 0.0.x Documents

The documents at `0.0.0` through `0.0.3` predate this framework and contain domain-specific content for category 0 (the addressing system itself). They are grandfathered — their addresses are valid and their content is authoritative. This document (`0.0.4`) is the first document in this section that is also a formal `.0` framework specification.

### 7.4 Migration Path

Existing nodes gain `.0` metadata by simply creating the files/directories. No changes to existing content are required. The process:

1. Node continues to work with frontmatter-only metadata (Tier 1)
2. When extended metadata is needed, create `N.0.md` or `N.0/` (Tier 2)
3. Add sub-sections as needed (Tier 3)

There is no "migration deadline." Nodes can remain frontmatter-only indefinitely.

## 8. Implementation Notes

### 8.1 Code Integration

The Hypernet Core (`0.1`) should recognize `.0` as a metadata sub-space:

- `Store.get_metadata(address)` — retrieves the `.0` index for a node
- `Store.put_metadata(address, section, data)` — writes to a specific `.0` sub-section
- `Node.metadata_ref` — property pointing to the node's `.0` space
- Metadata nodes have `object_type: "metadata-index"` or `object_type: "metadata-section"`

### 8.2 API Endpoints

The server should expose metadata operations:

- `GET /node/{address}/meta` — retrieve metadata index
- `GET /node/{address}/meta/{section}` — retrieve specific sub-section
- `PUT /node/{address}/meta/{section}` — update a sub-section
- `GET /node/{address}/meta/security` — shorthand for `.0.5`

### 8.3 Indexing

The metadata space should be indexed separately from content:
- Metadata nodes are excluded from content search by default
- A dedicated metadata search allows querying across `.0` spaces
- The type index should distinguish `metadata-index` and `metadata-section` types

### 8.4 Automation

Certain metadata sections can be auto-populated:
- `N.0.2` (versions) — auto-updated on node save
- `N.0.8` (analytics) — auto-updated on node access
- `N.0.3` (backups) — auto-updated on backup completion

Others require explicit creation:
- `N.0.1` (publishing) — set by the creator
- `N.0.5` (security) — set by the owner or administrator
- `N.0.4` (discussion) — created when a thread is started

## 10. Design Rationale

### Why .0 and not a separate metadata directory?

The `.0` convention means metadata lives *inside* the addressing hierarchy. It doesn't require a separate storage system, a separate database, or a separate directory tree. It's just more nodes at a reserved address. This follows the Hypernet's core principle: **the address IS the schema**.

### Why standardized sub-sections?

Without standard numbering, every node's metadata would be organized differently. Standard sub-sections mean that if you know where security lives for one node (`N.0.5`), you know where it lives for every node. This is navigability at scale.

### Why optional?

Most nodes don't need nine metadata sub-sections. A personal photo needs a visibility flag (frontmatter) and maybe an encryption reference (`.0.5`). A governance standard needs versions, discussion, security, and compliance tracking. The framework provides the structure without mandating the weight.

### Why recursive?

Because any rule that applies to "all nodes" must apply to metadata nodes too, or the system has a special case. The Hypernet has no special cases. A metadata node is a node. It can have metadata. This costs nothing when unused and provides unlimited depth when needed.

## Summary

The `*.0` Node Metadata Framework establishes:

1. **Universal reservation**: Every node's `.0` address is reserved for metadata
2. **Three tiers**: Frontmatter (always) → Index (on demand) → Extended (on demand)
3. **Nine standard sub-sections**: Publishing, Versions, Backups, Discussion, Security, Administration, Structure, Analytics, Extensions
4. **Infinite expandability**: Each sub-section can contain arbitrary depth; the recursive property allows meta-metadata
5. **Backward compatibility**: Existing nodes work unchanged; metadata is added incrementally
6. **Filesystem integration**: Maps cleanly to directories and files using existing naming conventions

This framework turns the reserved `.0` address from an informal convention into a structured, universal metadata system — one that can serve any need, at any depth, for any node in the Hypernet, now or in the future.
