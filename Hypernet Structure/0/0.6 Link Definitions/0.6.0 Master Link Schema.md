# 0.6.0 Master Link Schema

## Overview

Everything in the Hypernet is connected through Links. Links are first-class objects — not mere pointers between nodes, but rich, addressable, verifiable entities in their own right. A Link carries its own identity, metadata, access control, verification chain, and provenance history. Links are the connective tissue that transforms the Hypernet from a filing system into a knowledge graph.

This document defines the universal schema that all Links share, parallel to the Master Object Schema (0.5.0). Every Link type defined in 0.6.1 through 0.6.10 inherits from this schema.

## The Three Pillars

```
Objects (0.5.*)     Library (Addresses)     Links (0.6.*)
    store               organize               connect
 information           everything             everything
```

- **Objects** hold data.
- **The Library** addresses and locates.
- **Links** create meaning through relationships.

A document becomes valuable when linked to its author, sources, and related works. A person becomes part of a network when linked to colleagues, organizations, and contributions. An AI instance becomes trustworthy when linked to its verification chain, governance reviews, and contribution history.

## Core Link Structure

```yaml
link:
  # =====================================================================
  # IDENTITY (Required for all Links)
  # =====================================================================
  identity:
    address: "[0.6.x.y.instance]"       # Link's own HA in the 0.6.* space
    link_id: "[UUID]"                    # Globally unique identifier
    link_type: "[Type from 0.6.x]"      # e.g., "0.6.1.knows", "0.6.3.authored_by"
    relationship: "[Specific name]"      # e.g., "knows", "authored_by", "depends_on"
    version: "[Semantic version]"        # e.g., "1.0.0"

  # =====================================================================
  # ENDPOINTS (Required for all Links)
  # =====================================================================
  endpoints:
    source:
      address: "[Source Object HA]"      # e.g., "1.1" (Matt)
      object_type: "[0.5.x type]"        # e.g., "person"
      role: "[Source role label]"         # e.g., "author", "parent", "employer"

    target:
      address: "[Target Object HA]"      # e.g., "2.1" (Claude)
      object_type: "[0.5.x type]"        # e.g., "person", "document"
      role: "[Target role label]"         # e.g., "creation", "child", "employee"

  # =====================================================================
  # PROPERTIES (Link behavior and characteristics)
  # =====================================================================
  properties:
    # --- Directionality ---
    directed: true                       # true = source->target, false = bidirectional
    symmetric: false                     # If A->B implies B->A with same type
    transitive: false                    # If A->B and B->C implies A->C

    # --- Weight ---
    strength: 1.0                        # 0.0 to 1.0 confidence/weight
    weight_decay:                        # Optional: strength changes over time
      model: "none"                      # none | linear | exponential
      half_life_days: null               # For exponential decay

    # --- Cardinality ---
    cardinality: "many_to_many"          # one_to_one | one_to_many | many_to_one | many_to_many

    # --- Temporal Validity ---
    temporal:
      valid_from: "2026-03-07T00:00:00Z" # When this link becomes active
      valid_until: null                   # null = indefinite
      is_current: true                    # Computed: valid_from <= now < valid_until
      recurrence: null                    # Optional: cron pattern for recurring links

    # --- Ordering ---
    sort_order: null                      # For ordered relationships (e.g., author order)

  # =====================================================================
  # METADATA (Required for all Links)
  # =====================================================================
  metadata:
    created:
      timestamp: "2026-03-07T00:00:00Z"
      by: "[Creator HA]"                 # Who/what created this link
      method: "manual"                   # manual | import | inference | system

    modified:
      timestamp: "2026-03-07T00:00:00Z"
      by: "[Modifier HA]"

    context: "Specific context for this relationship"

    evidence:
      - type: "document"                 # document | assertion | inference | system
        reference: "[HA of evidence]"    # Points to supporting object
        confidence: 0.95                 # How strongly this evidence supports the link
      - type: "assertion"
        by: "[Person/AI who asserted]"
        confidence: 0.8

    tags: ["verified", "primary"]        # Freeform tags from 0.8.* flag system

    # --- Inverse Tracking ---
    inverse:
      relationship: "[Inverse name]"     # e.g., "authored_by" <-> "authored"
      auto_created: false                # Was the inverse auto-generated?
      inverse_link: "[HA of inverse]"    # Points to the inverse link object

  # =====================================================================
  # VERIFICATION (Trust and proof)
  # =====================================================================
  verification:
    status: "unverified"                 # See Verification Levels below
    verifiers:
      - entity: "[Verifier HA]"
        timestamp: "2026-03-07T00:00:00Z"
        method: "document_review"        # See Verification Methods below
        evidence: "[HA of verification evidence]"

    trust_score: 0.0                     # 0.0-1.0, computed from verification status
    dispute:
      contested: false                   # Has anyone disputed this link?
      contested_by: null                 # Who disputed
      reason: null                       # Why
      resolution: null                   # How resolved

  # =====================================================================
  # ACCESS CONTROL (Who can see/modify this link)
  # =====================================================================
  access:
    owner: "[Owner HA]"                  # Who controls this link
    visibility: "public"                 # public | restricted | private | endpoints_only
    permissions:
      read: "[Permission policy]"        # Who can see this link exists
      modify: "[Permission policy]"      # Who can change properties
      delete: "[Permission policy]"      # Who can soft-delete
      verify: "[Permission policy]"      # Who can add verification
    encryption:
      encrypted: false                   # Is this link's data encrypted?
      method: null                       # AES-256-GCM if encrypted
      key_reference: null                # Key envelope location

    # --- Consent ---
    consent:
      source_consented: true             # Source endpoint agreed to this link
      target_consented: true             # Target endpoint agreed to this link
      consent_required: "target"         # none | source | target | both
      consent_timestamp: "2026-03-07T00:00:00Z"

  # =====================================================================
  # PROVENANCE (History and audit trail)
  # =====================================================================
  provenance:
    history:
      - version: "1.0.0"
        timestamp: "2026-03-07T00:00:00Z"
        change: "Created"
        by: "[Creator HA]"
        previous_state: null             # Snapshot before change

    signatures:
      - signer: "[Signer HA]"
        signature: "[Cryptographic signature]"
        timestamp: "2026-03-07T00:00:00Z"
        covers: "full"                   # full | endpoints_only | properties_only

  # =====================================================================
  # LIFECYCLE
  # =====================================================================
  lifecycle:
    status: "active"                     # proposed | active | deprecated | archived | rejected
    proposed_by: null                    # If status=proposed, who proposed
    accepted_by: null                    # If accepted, who accepted
    deprecated_at: null                  # When deprecated
    deprecated_reason: null              # Why deprecated
    replacement: null                    # HA of replacement link if superseded
```

## Link Lifecycle

```
                    reject
               +-------------+
               |             |
               v             |
+---------+  propose  +----------+  accept  +--------+  deprecate  +------------+  archive  +----------+
| (none)  | -------> | Proposed | -------> | Active | ----------> | Deprecated | -------> | Archived |
+---------+          +----------+          +--------+             +------------+          +----------+
                                               |                                              |
                                               +----------------------------------------------+
                                                          (can be restored)

System-created links skip Proposed and go directly to Active.
```

### Status Definitions

| Status | Description | Graph Visibility |
|--------|-------------|-----------------|
| **proposed** | Created but awaiting consent from target endpoint | Hidden from traversal |
| **active** | Accepted and in use | Fully visible |
| **deprecated** | Superseded or outdated but preserved for history | Visible with warning |
| **archived** | Retained for provenance but not active | Hidden from traversal |
| **rejected** | Proposed but declined by target | Hidden, preserved for audit |

## Verification Levels

| Level | Trust Score | Description | Requirements |
|-------|------------|-------------|--------------|
| **unverified** | 0.0-0.2 | Just asserted, no confirmation | Link created |
| **self_attested** | 0.2-0.4 | Creator confirms | Creator verification |
| **mutual** | 0.4-0.6 | Both endpoints confirm | Source + target consent |
| **peer_verified** | 0.6-0.8 | Third party confirms | Independent verifier |
| **officially_verified** | 0.8-1.0 | Documentary/cryptographic proof | Evidence chain |

### Verification Methods

| Method | Description | Applicable To |
|--------|-------------|---------------|
| `self_attestation` | Creator declares the link | All links |
| `mutual_confirmation` | Both parties confirm | Person-person, org links |
| `document_review` | Evidence reviewed by verifier | Citations, authorship |
| `system_automatic` | System verifies programmatically | Timestamps, coordinates |
| `cryptographic` | Signed by both endpoints | High-security links |
| `community_consensus` | Multiple verifiers agree | Disputed or contested links |
| `ai_verification` | AI instance verifies consistency | Cross-reference checks |
| `provenance_chain` | Full chain of custody verified | Derived works, imports |

## Consent Model

Links between entities require consent based on the link type:

| Consent Mode | Description | Example |
|-------------|-------------|---------|
| **none** | No consent needed | System links, inferences |
| **source** | Source must consent | "I follow X" |
| **target** | Target must consent | "X is my friend" |
| **both** | Both must consent | Marriage, partnership |

Proposed links with `consent_required: "target"` or `"both"` enter the `proposed` lifecycle state and await acceptance. The target's owner can accept, reject, or ignore the proposal.

## Inverse Relationships

Many link types have natural inverses. The system tracks these:

| Link | Inverse | Auto-create? |
|------|---------|-------------|
| `authored_by` | `authored` | Yes |
| `parent_of` | `child_of` | Yes |
| `reports_to` | `manages` | Yes |
| `depends_on` | `dependency_of` | Yes |
| `employed_by` | `employs` | Yes |
| `cites` | `cited_by` | Yes |
| `follows` | `followed_by` | No (asymmetric) |
| `trusts` | `trusted_by` | No (asymmetric) |

When `auto_create` is true, creating a link automatically creates the inverse. Both links reference each other through the `inverse.inverse_link` field.

## Transitivity

Some links are transitive: if A->B and B->C, then A->C can be inferred.

| Transitive Links | Example |
|-----------------|---------|
| `reports_to` | A reports to B, B reports to C => A indirectly reports to C |
| `subsidiary_of` | A subsidiary of B, B subsidiary of C => A subsidiary of C |
| `derived_from` | A derived from B, B derived from C => A derived from C |
| `contains` (spatial) | Room contains desk, desk contains laptop => room contains laptop |
| `preceded_by` | C preceded by B, B preceded by A => C preceded by A |

Transitive links are **not auto-created** but can be **inferred on query** with configurable max depth.

### Index Structures

Links maintain bidirectional indexes for fast traversal:

| Index | Purpose | Lookup |
|-------|---------|--------|
| `links_from` | Outgoing links from source | source_address -> [link_ids] |
| `links_to` | Incoming links to target | target_address -> [link_ids] |
| `links_by_type` | Links by relationship type | relationship -> [link_ids] |
| `links_by_status` | Links by lifecycle status | status -> [link_ids] |
| `links_pending` | Proposed links awaiting consent | target_address -> [link_ids] |

## Validation Rules

All Links must pass validation:

```yaml
validation:
  required_fields:
    - identity.link_type
    - identity.relationship
    - endpoints.source.address
    - endpoints.target.address
    - metadata.created.timestamp
    - metadata.created.by
    - lifecycle.status

  constraints:
    - "source != target (no self-links)"
    - "strength in range [0.0, 1.0]"
    - "link_type matches a defined 0.6.* category"
    - "source_type matches link type's allowed source_types"
    - "target_type matches link type's allowed target_types"
    - "cardinality constraints satisfied"
    - "consent requirements met before status=active"

  type_validation:
    method: "link_type_schema"
    schema_location: "0.6.[category]"
```

## Graph Operations

The link network enables:

| Operation | Description | Complexity |
|-----------|-------------|-----------|
| **Traversal** | Follow links to discover related objects | O(E) per hop |
| **Pathfinding** | Find shortest/all paths between nodes | BFS/DFS |
| **Clustering** | Identify groups of densely connected objects | Community detection |
| **Inference** | Derive new links from transitivity/patterns | Rule-based |
| **Recommendation** | Suggest relevant content based on link patterns | Collaborative filtering |
| **Impact Analysis** | Trace consequences through dependency chains | Forward traversal |
| **Provenance Trace** | Follow creation/derivation chain to origin | Backward traversal |
| **Trust Propagation** | Compute trust scores through trust link chains | Weighted traversal |

## Link Addressing Convention

Links are addressable objects. Their addresses follow this pattern:

```
0.6.[category].[type].[instance]

Examples:
  0.6.1.knows.00001          -> First "knows" link
  0.6.3.authored_by.00042    -> 42nd authorship link
  0.6.8.instance_of.00001    -> First AI instance link
```

When a link is embedded in an object's frontmatter (e.g., in a .md file), it references the link type but doesn't need its own instance address — the link is stored in the link index, not as a standalone file.

## Comparison: Objects vs Links

| Feature | Objects (0.5) | Links (0.6) |
|---------|--------------|-------------|
| Identity | address, object_id, type | address, link_id, type, relationship |
| Endpoints | N/A | source + target (with types) |
| Metadata | created, modified, status | created, modified, context, evidence |
| Access | owner, permissions, encryption | owner, permissions, encryption, consent |
| Content | Type-specific payload | Properties (direction, weight, temporal) |
| Verification | N/A (via links) | Multi-level verification chain |
| Provenance | History, signatures | History, signatures |
| Lifecycle | Created->Active->Deprecated->Archived | Proposed->Active->Deprecated->Archived |
| Flags | From 0.8.* | From 0.8.* (via tags) |

Links match Objects in every structural dimension and add consent and verification as Link-specific features.
