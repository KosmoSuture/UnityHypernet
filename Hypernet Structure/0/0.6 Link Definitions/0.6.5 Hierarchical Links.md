# 0.6.5 Hierarchical Links

## Overview

Hierarchical links define structural relationships — how objects nest within each other, how taxonomies organize knowledge, and how complex systems decompose into parts. These are the links that give the Hypernet its tree-like navigability while preserving the full graph underneath.

## Structural Links

### parent_of / child_of
Direct hierarchical relationship.

```yaml
link_type:
  name: "parent_of"
  address: "0.6.5.parent_of"
  category: "hierarchy"
  directed: true
  symmetric: false
  transitive: false    # parent_of is NOT transitive (grandparent != parent)

  source_types: ["*"]  # Any object can be a parent
  target_types: ["*"]  # Any object can be a child
  cardinality: "one_to_many"  # One parent, many children

  properties:
    order: "integer"           # Sort order among siblings
    relationship: "structural" # structural | logical | temporal

  inverse: "child_of"

  verification:
    method: "system_automatic"  # Derived from address hierarchy
    can_override: true          # Manual links can override address-derived hierarchy

  notes: |
    The Hypernet address system (1.1 -> 1.1.1 -> 1.1.1.1) implies a natural
    parent-child hierarchy. These links make that hierarchy explicit and queryable.
    Additional parent_of links can create logical hierarchies that differ from
    the address-based physical hierarchy.
```

### part_of / has_part
Component-whole relationship.

```yaml
link_type:
  name: "part_of"
  address: "0.6.5.part_of"
  category: "composition"
  directed: true
  symmetric: false
  transitive: true     # If A part_of B, B part_of C, then A part_of C

  source_types: ["*"]
  target_types: ["*"]
  cardinality: "many_to_many"  # A part can belong to multiple wholes

  properties:
    part_type: "component|section|chapter|module|element|fragment"
    essential: "boolean"       # Is this part essential to the whole?
    quantity: 1                # How many of this part
    position: "string"        # Where within the whole

  inverse: "has_part"

  verification:
    method: "structural_analysis"

  constraints:
    no_cycles: true            # A cannot be part_of itself through any chain
```

### contains / contained_in

```yaml
link_type:
  name: "contains"
  address: "0.6.5.contains"
  category: "containment"
  directed: true
  symmetric: false
  transitive: true

  source_types: ["*"]

  cardinality: "one_to_many"

  properties:
    containment_type: "logical|physical|virtual|administrative"
    exclusive: "boolean"       # Is the contained object in only this container?

  inverse: "contained_in"

  verification:
    method: "system_automatic"
```

## Taxonomic Links

### broader_than / narrower_than
Taxonomic breadth relationship.

```yaml
link_type:
  name: "broader_than"
  address: "0.6.5.broader_than"
  category: "taxonomy"
  directed: true
  symmetric: false
  transitive: true

  source_types: ["concept", "category"]
  target_types: ["concept", "category"]
  cardinality: "one_to_many"

  properties:
    taxonomy: "[Taxonomy HA]"  # Which taxonomy system
    level_difference: 1        # How many levels apart

  inverse: "narrower_than"

  verification:
    method: "taxonomy_validation"

  notes: |
    Examples:
    - "Programming Language" broader_than "Python"
    - "Animal" broader_than "Mammal" broader_than "Dog"
    - "Hypernet System (0.*)" broader_than "Link Definitions (0.6.*)"
```

### instance_of / has_instance
Type-instance relationship.

```yaml
link_type:
  name: "instance_of"
  address: "0.6.5.instance_of"
  category: "classification"
  directed: true
  symmetric: false
  transitive: false    # An instance of an instance is not an instance of the original

  source_types: ["*"]
  target_types: ["concept", "category"]
  cardinality: "many_to_one"  # Each instance has one primary type

  properties:
    classification_source: "manual|inferred|imported"
    confidence: "0.0-1.0"     # For inferred classifications

  inverse: "has_instance"

  verification:
    method: "type_validation"

  notes: |
    Examples:
    - "Matt (1.1)" instance_of "Person (0.5.1)"
    - "This document" instance_of "Markdown Document (0.5.3.1)"
    - "Claude (2.1)" instance_of "AI Entity"
```

Schema/type inheritance.

```yaml
link_type:
  name: "inherits_from"

  category: "inheritance"
  directed: true
  symmetric: false
  transitive: true

  source_types: ["concept", "document"]

  cardinality: "many_to_many"  # Multiple inheritance allowed

  properties:
    inheritance_type: "full|partial|interface"
    overrides: ["list of overridden properties"]

  inverse: "inherited_by"

  verification:
    method: "schema_analysis"

  notes: |
    Examples:
    - "Person Object (0.5.1)" inherits_from "Master Object (0.5.0)"
```

## Hierarchy Properties

### Depth and Breadth Metrics

Every hierarchical link contributes to computable metrics:

| Metric | Description | Example |
|--------|-------------|---------|
| **Depth** | Distance from root | 1.1.1.1.00001 = depth 5 |
| **Breadth** | Number of siblings at same level | 5 children of 1.1 |
| **Fan-out** | Total descendants | All nodes under 2.1.* |
| **Fan-in** | Total ancestors/containers | All groups containing node X |
| **Height** | Max depth of subtree below | Deepest descendant |

### Hierarchy Validation Rules

```yaml
validation:
  rules:
    - "No cycles in parent_of chains"
    - "No cycles in part_of chains"
    - "No cycles in contains chains"
    - "instance_of targets must be type definitions (0.5.* or concepts)"
    - "inherits_from must not create diamond problems without explicit resolution"
    - "broader_than/narrower_than must be within same taxonomy"
```
