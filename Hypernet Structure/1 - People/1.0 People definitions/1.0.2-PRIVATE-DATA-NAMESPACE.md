# 1.0.2 — The `*.private` Namespace Pattern

## Origin

Per Matt directive (2026-05-01):

> Add the `1.1.private`. I kind of like that in general. We can do
> a lot with that, including storing keys and other cryptographic
> identity. DNA as well. Anything that tries to access this data
> needs to go through extra scrutiny to make sure it's legitimate.

This document promotes the existing `1.1 Matt Schaeffer/private/`
folder pattern into a general convention for any account on the
Hypernet that needs a private-data namespace.

## The Pattern

Any account `X` (human `1.X`, AI `2.X`, business `3.X`, knowledge
`4.X`) MAY define a private namespace at:

- **Address**: `X.private`
- **Folder**: a `private/` subdirectory under that account's root
- **Gitignore status**: explicitly excluded from public repository
  via the `**/private/` pattern in root `.gitignore`

The `X.private` address is **resolvable** (so other Hypernet
participants can reason about its existence and request access to
it), but its **contents** are not publicly addressable. The
namespace is the public surface; the content underneath is
private.

## What Belongs in `*.private`

This namespace is for data that is sensitive enough that even its
*existence-by-name* is private to the account owner, and where any
external request to read or write it must pass extra scrutiny.

Examples:

- **Cryptographic identity material** — private keys for the
  account's signing key, transport key, registry attestation key,
  PGP private keys, SSH keys, client certificates
- **OAuth tokens and credentials** — Gmail/Dropbox/OneDrive refresh
  tokens, API keys for third-party services, password vaults
- **Biometric / biological identity** — DNA sequence data, retina
  prints, fingerprint hashes, voice prints, facial geometry
- **Personal master records** — pre-aliased master account data
  before it is exposed through aliases or lockers
- **Raw imports staging** — unprocessed data dumps from email,
  social media, cloud storage that have not yet been triaged into
  the public Hypernet structure
- **Encrypted documents at rest** — GPG-encrypted personal
  archives, sealed legal documents, sealed medical records
- **Recovery seeds and break-glass material** — wallet seed
  phrases, account recovery keys, executor instructions

The unifying property: this is data the account owner would want
**zero ambient access** to, even by node operators or AI helpers
under that account, unless a specific authorized request goes
through the verification flow below.

## Extra-Scrutiny Access Flow

Any read or write attempt against `X.private.*` MUST traverse this
flow:

1. **Identity verification.** The requester (app, AI helper,
   external party) must present a verifiable identity. For AI
   helpers, that means a `0.5.17` boot sequence with hash
   matching the Official registry. For apps, a `0.5.18` app load
   with attested hash. For humans, signed credentials from a
   trusted authority.
2. **Authorization check.** The account owner's access policy is
   queried for an explicit grant covering this address prefix.
   No ambient grant on `X.*` covers `X.private.*` — `private`
   must be named explicitly.
3. **Security-AI review.** If the account owner has configured a
   security AI (per `personal-ai-swarm.md`), the request is
   inspected by that AI before any grant resolves. The security
   AI evaluates request shape, requested scope, declared reason,
   and time-of-day / context anomalies. It can deny, narrow, or
   escalate to the owner.
4. **Out-of-band confirmation for high-risk scopes.** Reads or
   writes against keys, biometric data, or recovery material
   require an out-of-band confirmation from the owner — a phone
   approval, a hardware key tap, or a multi-channel sign-off.
   Single-channel approval is insufficient.
5. **Audit log entry.** Every attempt — granted, denied, or
   escalated — appends an entry to the account's audit log. The
   log entry is itself an addressable Hypernet object so it
   cannot be silently scrubbed.
6. **Rate limiting + anomaly detection.** Multiple `X.private`
   access attempts within a short window trigger a soft-lock and
   notify the owner.

This is "extra scrutiny." The default access-policy enforcement
applies to all addresses; `*.private` adds these layers on top.

## Relationship to Lockers, Mandalas, and Aliases

- **Aliases (`X.alias.<name>`)** are the *public* surface a person
  presents.
- **Lockers (under `X.locker.<name>`)** are *access-scoped link
  manifests* — collections of pointers to data the owner has
  decided can be granted to specific recipients under specific
  conditions.
- **Mandalas** are the *access patterns* that open lockers.
- **Private namespace (`X.private`)** is the *base material* —
  what lockers ultimately point at, what mandalas ultimately
  decrypt fragments of. Most data starts in `X.private` and
  becomes selectively visible by being linked from a locker that
  a mandala can open.

The privacy ladder, summarized:

```
X.private (base material — extra scrutiny on every access)
   ↑
X.locker.<name> (link manifests — opened by mandalas)
   ↑
X.alias.<name> (public-facing identity surface)
   ↑
public-facing data (auto-granted public mandalas)
```

## Cryptographic Material Specifically

Per Matt's directive, `X.private` is the canonical home for
cryptographic identity:

- **Account signing keys** — the private half of the keypair the
  account uses to sign claims, attestations, and outgoing
  Hypernet messages
- **Node operator keys** (for `1.X` accounts that operate Official
  nodes) — node signing key, transport key, registry attestation
  key per `0.2.6 Official Registry`
- **Encryption-at-rest keys** — keys used to envelope-encrypt
  data in lockers per `0.5.0` access.encryption
- **Recovery / break-glass material** — sealed copies of master
  keys with multi-party reconstruction conditions

Cryptographic material in `X.private` SHOULD additionally:

- Be encrypted at rest with a key derived from a hardware module
  (TPM, Secure Enclave, YubiKey, similar)
- Never be exposed in plaintext to AI helpers — AI helpers may
  *invoke* signing operations through a security-AI-mediated
  signing service, never *read* the key bytes
- Have rotation schedules documented in the account's audit log

## Biometric / DNA Specifically

Biometric data is unusually sensitive because it is non-revocable
— a leaked fingerprint cannot be re-issued. `X.private.biometric.*`
treatment:

- Stored as a *one-way derivation* (hash, fuzzy hash, or template
  vector) wherever feasible — not raw images
- DNA sequence data MAY be stored in raw form for medical /
  research utility, but only with the strictest extra-scrutiny
  flow
- Any external request for biometric data is escalated to the
  owner regardless of prior grants
- Companion AI helpers do not have ambient access to biometric
  data even if they have ambient access to other `X.*` regions

## Open Questions for Iteration

- **Inter-account private exchange.** When two accounts need to
  privately share data (e.g., a couple with a shared family
  locker), do they create a third joint account with its own
  `*.private`, or do they cross-grant from each personal
  `*.private` via mandala? Probably a future joint-account
  feature.
- **Recovery without single point of failure.** Shamir-secret-
  shared recovery across trusted contacts vs. a designated
  custodian — which becomes the default?
- **DNA-specific governance.** DNA data has familial implications
  (a person's DNA reveals information about their relatives).
  Storing DNA in `X.private` may need an additional consent layer
  for relatives whose information is implicitly disclosed.
- **Hardware module standardization.** Which TPM / Secure Enclave
  / YubiKey APIs does the runtime target first?

These are noted for future iteration, not blocking for the v1
specification.
