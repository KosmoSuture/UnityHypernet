# 1.0.1 - Lockers, Mandalas, And Aliases For People

## Purpose

This document defines the starting privacy framework for human `1.*` accounts.
People need a basic default structure, but every person must be able to modify
their own account over time.

The default rule is:

**The master `1.*` account is a private home. Public interaction happens through
aliases and lockers, not by browsing the master account directly.**

## Core Definitions

### Master Account

The master `1.*` account is the person's private address space. It contains the
canonical data and links for that person.

The master account should be difficult to address directly from public surfaces:

- public search should prefer aliases, public profiles, and public lockers;
- direct master-address reads should require owner authorization except for
  deliberately public metadata;
- the user and their trusted AI assistant hold the direct encrypted access path;
- public messages and public access requests should go through aliases or
  locker request routes.

### Locker

A locker is an access-scoped link manifest.

It does **not** contain the private data payload. It contains links back to data
objects, documents, media, records, claims, or other nodes in the private
account.

The locker is the maximum data set that one key can open. Access to another
locker requires a separate key or grant.

Example locker categories:

- financial locker: identity, address, tax, income, payment, and bank-form data
- social locker: posts, public-facing photos, videos, friend-visible material
- medical locker: medical history, medications, allergies, conditions,
  emergency contacts, insurance, physician notes, and device records
- professional locker: resume, portfolio, certifications, work history,
  professional contact routes
- family locker: family photos, household records, delegated permissions
- credentials locker: password/secret references and recovery metadata

### Mandala

A mandala is the access pattern that opens a locker.

It defines:

- who can see the locker;
- which linked records inside the locker can be seen;
- whether non-granted records are invisible or only redacted;
- whether the grantee can read, write, append, verify, or request changes;
- duration, expiration, revocation, and audit requirements;
- emergency and break-glass rules when applicable.

The most important rule:

**A mandala can only reveal the data it grants. Non-allowed data is invisible,
including existence, count, title, metadata, and graph neighborhood unless the
mandala explicitly grants a redacted existence view.**

### Alias

An alias is an outward-facing identity or context.

A person can have many aliases:

- public real-name profile;
- professional alias;
- creative/artist identity;
- social handle;
- anonymous or pseudonymous participation identity;
- emergency medical access identity;
- business-facing contact identity.

Aliases may be internally linked to the master account, but public users must
not be able to resolve that link unless the owner explicitly publishes it.

Category `9 - Aliases` currently describes aliases as public pointers to primary
nodes. Human privacy requires a stricter form:

- public aliases can resolve to public profiles or public lockers;
- private aliases can map back to the master `1.*` account only inside the
  owner's encrypted/private graph;
- an alias may be verifiable without being deanonymizing.

## Visibility Model

| Surface | Public can see? | Notes |
|---|---:|---|
| Master `1.*` account root | No, except safe public metadata | Treat as private home, not public storefront |
| Public profile alias | Yes | Owner controls what is published |
| Locker existence | Only if public or requestable | Sensitive lockers can be invisible |
| Locker contents | No | Locker contains links, not payloads |
| Mandala grant | Only grant-specific metadata | Public mandalas can be auto-granted |
| Linked private data | Only when mandala grants it | Non-granted data does not exist to requester |
| Alias to master mapping | No by default | Internal-only unless owner publishes link |

## Locker Mechanics

A locker should be modeled as a typed object with a link list:

```yaml
locker:
  id: "locker-medical-primary"
  owner: "1.X"
  category: "medical"
  visibility: "requestable"        # public | requestable | private | hidden
  link_manifest:
    - target: "1.X.6.medical.allergies"
      label: "Allergies"
      required_scope: "emergency-read"
    - target: "1.X.6.medical.medications"
      label: "Current medications"
      required_scope: "emergency-read"
    - target: "1.X.6.medical.full-history"
      label: "Full medical history"
      required_scope: "physician-read"
  mandalas:
    - "mandala-medical-emergency"
    - "mandala-primary-care"
  audit_required: true
```

The payload records remain at their canonical addresses. The locker is a
permissioned map of what can be opened together.

## Mandala Mechanics

A mandala should be modeled as a grant policy:

```yaml
mandala:
  id: "mandala-medical-emergency"
  owner: "1.X"
  opens_locker: "locker-medical-primary"
  grants:
    read:
      - "1.X.6.medical.allergies"
      - "1.X.6.medical.medications"
      - "1.X.6.medical.conditions"
      - "1.X.6.medical.emergency_contacts"
  grantee_requirements:
    verified_role: "licensed_medical_professional"
    context: "emergency"
    attestation_required: true
  visibility_mode: "non_granted_invisible"
  expires_after: "24h"
  audit:
    log_access: true
    notify_owner: true
    notify_trusted_assistant: true
    require_reason: true
  revocation:
    owner_can_revoke: true
    auto_revoke_on_expiration: true
```

## Public Data

Truly public data still goes through lockers and mandalas.

The difference is that the mandala is automatically granted:

- anonymous public reader mandala;
- public profile mandala;
- public project mandala;
- public key and claim-verification mandala.

This keeps the access model uniform. Public data is not outside the privacy
system; it is data whose mandala has an open grant.

## Emergency Medical Locker

The medical locker is the clearest high-value example.

A person may publish an emergency pointer through:

- medical bracelet;
- ID barcode;
- NFC phone tag;
- wallet card;
- emergency contact record.

The pointer should resolve only to an emergency locker request route, not to the
master account.

Emergency flow:

1. Doctor or emergency responder scans the pointer.
2. System presents medical emergency access request.
3. Requester verifies qualified medical role where possible.
4. Requester attests emergency context.
5. Mandala grants the emergency medical scope.
6. System logs all accessed links and notifies owner/trusted assistant.
7. Grant expires automatically.

The emergency mandala can grant broad access when the owner configured it that
way, including full medical history. That grant should still be audited,
time-limited, and bound to emergency purpose.

## Alias And Locker Examples

### Professional Alias

```text
Public identity: 9.professional.alex-architect
Public surface: professional profile locker
Master link: hidden internal link to 1.X
Visible data: resume, certifications, portfolio, public work history
Invisible data: legal name, family, private finances, medical, personal media
```

### Social Alias

```text
Public identity: 9.social.creator-name
Public surface: social locker
Master link: hidden internal link to 1.X
Visible data: selected posts, photos, videos, follower inbox policy
Invisible data: private address, family graph, financial records, medical data
```

### Public Real-Name Profile

```text
Public identity: 1.X.12 or published alias
Public surface: public profile locker
Visible data: owner-approved profile, public keys, public claims
Invisible data: anything not linked through public mandala
```

## Enforcement Requirements

This model requires read-time graph enforcement:

1. Resolve requester identity and context.
2. Resolve alias or locker request route.
3. Evaluate mandala grants.
4. Build a visible subgraph containing only granted links and granted targets.
5. Hide non-granted data and non-granted graph edges.
6. Audit access where required.

The important implementation principle is **visibility by constructed subgraph**,
not visibility by filtering after a full graph is already exposed.

## Relationship To Existing Structures

- `1.X.6 - Personal Data` remains the canonical personal data area.
- `1.X.11 - Secrets & Credentials` stores credential metadata and encrypted
  references only.
- `1.X.12 - Public Profile & Lockers` is the browsable public surface.
- `9 - Aliases` remains the global alias namespace, but private aliases must not
  publicly resolve to master accounts.
- Core access policy must eventually enforce mandalas at object and link read
  time.

## Open Questions

1. Should hidden lockers be discoverable by emergency responders if the pointer
   is presented, or only by exact emergency route?
2. Should public aliases live entirely in `9.*`, or should each `1.*` account
   also expose local aliases under its public locker surface?
3. How should a person prove that two aliases are linked without revealing the
   master account?
4. What minimum medical fields belong in an emergency default?
5. What is the safest owner-recovery path if the direct encrypted key is lost?
