# Personal Assistant App - Engineering Plan (Caliper half)

This is the engineering companion to Keel's design and MVP scoping
documents:

- `2026-05-02-personal-assistant-app-design.md`
- `2026-05-02-personal-assistant-app-mvp.md`

Keel defined the relationship shape and minimum useful product.
This document defines the buildable system boundary, stack,
interfaces, data addresses, app-load linkage, and acceptance
criteria. It is a plan, not built code.

## Product Boundary

The app is a personal Hypernet companion surface for Matt's
assistant. It is not a generic productivity app and not an
unbounded autonomous agent. It gives Matt a reliable way to talk
to his assistant throughout the day, file the conversation into
Hypernet addresses, surface the top priority, and connect the
assistant to carefully granted personal data.

The MVP should prove one loop:

1. Matt talks to the assistant from Android.
2. The assistant answers with context from the Hypernet archive.
3. The conversation is logged as an addressable object.
4. Calendar read data improves the answer.
5. The watch can surface one top-priority item.
6. Every permission and data movement can be inspected.

## Architecture

```text
Android app
  - push-to-talk voice surface
  - transcript and playback
  - local permission UI
  - local encrypted cache

Wear OS companion
  - top-priority tile
  - notification actions
  - phone data-layer bridge

Hypernet assistant backend
  - FastAPI service
  - auth and app-load verification
  - conversation turn endpoint
  - briefing endpoint
  - connector sync endpoint
  - priority engine endpoint

Connector layer
  - Google Calendar read-only first
  - Gmail and Tasks later
  - OAuth token vault
  - per-connector audit records

Archive layer
  - addressable conversation markdown
  - structured events
  - links to projects, calendar events, approvals
  - encrypted local/private storage where required

AI runtime
  - assistant boot/personality context
  - process-load retrieval
  - tool calls through declared app-load scopes
  - security-AI review for private-data expansions
```

## Recommended Stack

| Layer | Recommendation | Reason |
|---|---|---|
| Phone app | Kotlin + Jetpack Compose | Native Android, modern UI, good permissions model |
| Watch app | Wear OS Tiles + Data Layer API | Matches Galaxy Watch, keeps MVP glance-focused |
| Local storage | Room + SQLCipher or Android Keystore-backed files | Offline cache without plaintext private data |
| Background work | WorkManager | Calendar sync, morning briefing, retryable jobs |
| Voice capture | Android AudioRecord / MediaRecorder in explicit sessions | Reliable push-to-talk and meeting-mode foundation |
| Wake word | Defer; later evaluate Picovoice or Android on-device hotword options | Push-to-talk reduces MVP risk and privacy concern |
| STT | Local-first where practical; API fallback only with explicit approval | Keeps privacy posture honest |
| TTS | Android system TTS first; higher-quality API later by consent | Fast MVP, no lock-in |
| Backend | Existing Hypernet FastAPI patterns | Reuses current codebase and test style |
| Models | Provider-pluggable assistant runtime | Assistant identity must survive model changes |
| OAuth | Google Calendar read-only first | Proves useful throughout-day context with low blast radius |

## Address Model

The app must write addressable records from the first build. Draft
addresses:

| Record | Address pattern |
|---|---|
| App load | `0.5.18.1.1` |
| Conversation log | `1.1.10.1.conversations.<yyyymmdd-hhmmss>.<slug>` |
| Voice session event | `1.1.10.1.app.personalassistant.sessions.<yyyymmddhhmmss>` |
| Morning briefing | `1.1.10.1.app.personalassistant.briefings.<yyyymmdd>` |
| Priority item | `1.1.10.1.app.personalassistant.priorities.<id>` |
| Calendar connector sync | `1.1.10.1.app.personalassistant.connectors.googlecalendar.<yyyymmddhhmmss>` |
| Permission prompt | `1.1.10.1.app.personalassistant.permissions.<id>` |
| Watch notification | `1.1.10.1.app.personalassistant.notifications.<id>` |

Conversation logs should link to:

- the assistant identity that answered;
- the app-load instance in effect;
- the source calendar event, if any;
- the project, task, or locker the conversation belongs to;
- any permission prompt that expanded access during the session.

## Backend API Surface

Initial endpoints should stay small:

| Endpoint | Method | Purpose |
|---|---|---|
| `/assistant/v1/session/start` | POST | Create an addressable voice/chat session |
| `/assistant/v1/session/{id}/turn` | POST | Submit transcript text/audio reference and receive assistant response |
| `/assistant/v1/session/{id}/close` | POST | Finalize transcript, summary, links, and filing |
| `/assistant/v1/briefing/today` | GET | Return morning/top-of-day briefing |
| `/assistant/v1/priorities/current` | GET | Return the current watch tile item |
| `/assistant/v1/notifications/{id}/ack` | POST | Record dismiss/snooze/ask-now action |
| `/assistant/v1/connectors/google-calendar/sync` | POST | Sync calendar data under declared OAuth scope |
| `/assistant/v1/permissions/request` | POST | Create a mandala/security-AI review request |

MVP can accept transcript text from the phone first and add audio
upload/reference later. Audio blobs should not be treated as the
canonical record; the canonical record is the addressable session
object containing summary, transcript, model metadata, and links.

## Security and Permission Rules

The app should be governed by the `0.5.18.1.1` app-load instance.
Before private data is accessed, Matt or a user-owned security AI
should be able to answer:

- What exact app-load is running?
- Is it Official, private fork, or local dev?
- Which account, locker, connector, or address prefix is being read?
- What will be written?
- Where is the audit log?
- Can this grant be revoked without losing the audit trail?

Default denials:

- no direct scan of the master `1.*` account;
- no hidden always-on recording;
- no email sending in MVP;
- no calendar write until the read-only loop is stable;
- no private locker expansion without a visible permission prompt;
- no model/provider switch that changes data handling without a new audit event.

Always-on transcription is not an MVP capability. The architecture
should support future scoped modes:

- push-to-talk: explicit user action, default MVP;
- meeting mode: explicit start/stop, visible indicator, auto-timeout;
- drive mode: voice-only interaction, no full passive transcription;
- experimental always-listening: off by default, visibly marked,
  audit-heavy, and not eligible for Official default mode until
  reviewed.

## Build Sequence

### Phase 0 - Repo Prep

- Create app-load instance `0.5.18.1.1`.
- Define Pydantic request/response schemas for sessions, turns,
  priorities, briefings, permission prompts, and connector syncs.
- Add filesystem writer for addressable conversation logs.
- Add a dev-only local assistant backend route.

Exit: a test can create a session, write a markdown conversation
object, and link it to the app-load.

### Phase 1 - Android Push-to-Talk

- Build Android shell with one talk button.
- Capture audio, transcribe, send transcript text to backend.
- Play assistant response through Android TTS.
- Show transcript and session summary.

Exit: Matt can have a one-turn and multi-turn conversation by
pressing a button, and the archive record is created.

### Phase 2 - Calendar Read

- Add Google OAuth read-only calendar scope.
- Sync today's events to the connector layer.
- Include calendar context in `what's on my plate?`.
- Record sync events and access reasons.

Exit: the assistant can answer today's schedule from live calendar
data and the user can inspect what was read.

### Phase 3 - Morning Briefing

- Add configurable briefing time.
- Use WorkManager for local scheduling.
- Backend produces a three-item briefing from calendar, manually
  flagged items, and open approval queue.

Exit: one daily notification opens or speaks the briefing.

### Phase 4 - Wear OS Tile

- Add a Wear OS top-priority tile.
- Phone pushes the current priority via the Data Layer API.
- Watch supports dismiss, snooze, and ask-now actions.

Exit: the watch can show one useful priority without opening the
phone app.

### Phase 5 - Calendar Write With Confirmation

- Add calendar write scope only after explicit upgrade.
- Require spoken or tapped confirmation before write.
- Store before/after event details in the audit record.

Exit: the "First Real Conversation" test from Keel's MVP doc can
pass end-to-end.

### Phase 6 - Meeting Mode and Wider Connectors

- Add explicit meeting mode with visible indicator and timeout.
- Add Gmail and task connectors as separate app-load revisions.
- Add project filing suggestions and review queue.

Exit: the app begins moving from MVP companion to the full
"rudder beneath the waters" vision.

## Testing Strategy

Backend tests:

- app-load scope validation;
- session create/turn/close lifecycle;
- conversation file frontmatter and address format;
- calendar connector dry-run with fixture data;
- permission denial for undeclared scopes;
- audit record creation for every connector read/write.

Android tests:

- mic permission denial path;
- push-to-talk state machine;
- transcript display and retry;
- lost-network queued session;
- local cache encryption smoke test.

Wear OS tests:

- tile shows current priority;
- stale data state is visible;
- dismiss/snooze actions reach phone/backend;
- no dense private text displayed on locked watch.

Acceptance test:

Use Keel's "First Real Conversation" test as the release gate.
If Matt cannot ask what his morning looks like, add a reminder
with confirmation, and receive the watch reminder, the MVP has
not shipped.

## Implementation Risks

- Android background execution restrictions can make always-on or
  near-always-on behavior fragile. Keep MVP user-initiated.
- Wear OS tiles are intentionally simple. Do not overload them.
- OAuth consent screens and token storage can become a security
  distraction. Start with Calendar read-only.
- Speech reliability will determine whether Matt uses the app.
  A rough UI with reliable voice is better than polished screens
  with unreliable speech.
- The assistant identity must not collapse into whichever model is
  called. Log model/provider metadata, but keep the assistant
  archive and role separate from the model runtime.

## Current Status

Draft plan with Phase 0 backend scaffold landed on 2026-05-02.
The code now includes:

- `hypernet/assistant_app.py` for session lifecycle, app-load
  write-scope validation, Store node writes, and markdown
  conversation logs;
- a Phase 0.5 model-adapter seam plus deterministic local responder
  for test/dev auto-responses without provider secrets;
- FastAPI routes under `/assistant/v1/session/*` and
  `/assistant/v1/app-load/check-scope`, plus
  `/assistant/v1/sessions` for resume/list views;
- test coverage in `test_personal_assistant_app_phase0`.

This is not the Android app, Wear OS app, live AI runtime, or
calendar connector yet. The next engineering task is Phase 1:
Android push-to-talk client integration against these backend
routes, or a provider-backed Phase 0.6 adapter if we want live
model calls before mobile work starts.
